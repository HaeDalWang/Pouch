# 🦦 pouch

> **Pouch — your own otter's pouch of tools. One day, a whole raft of them.**

해달은 가슴팍 겨드랑이에 느슨한 살 주머니(pouch)를 가지고 다닌다. 그 안엔 무리가 다 같이 쓰는 똑같은 돌이 아니라, 평생 아끼는 **나한테 맞는 단 하나의 돌**이 들어있다. 그 돌로 조개를 깨다 보면 돌도 손에 맞게 닳아간다.

`pouch`는 에이전트에게 도구와 스킬을 쥐어주는 **개인 하네스(harness)**다. 남이 만든 정답 세트를 받아 쓰는 게 아니라, 설치 마법사가 나에게 맞는 도구를 골라 주머니에 넣어주고, 쓸수록 나한테 맞게 닳아가는 — 그런 도구를 목표로 한다.

---

## 이걸 쓰면 뭐가 달라지나

매번 같은 에러를 세 번씩 복붙해서 설명한 적 있나요? 새 세션마다 "나는 DevOps고,
python 주로 쓰고, 이 프로젝트엔 이런 규칙이 있고…"를 다시 읊은 적은요?

에이전트는 세션이 끝나면 나를 잊는다. 그래서 매번 처음부터 다시 설명하게 된다.
**pouch는 그 반복을 없앤다** — 한 번 알려준 것(나에 대한 것, 되풀이되는 에러의
해법, 넘지 말아야 할 선)을 기억해뒀다가, 새 세션 첫머리에 에이전트에게 대신
넣어준다.

| 없으면 겪는 일 | pouch가 있으면 |
| --- | --- |
| 매 세션 나를 처음부터 다시 설명 | 나에 대한 기억이 자동으로 먼저 들어간다 |
| 같은 에러 해법을 매번 다시 알려줌 | 한 번 적어두면 다음 세션이 알고 시작 |
| 스킬·플러그인 수백 개에 도구 목록이 파묻힘 | 실제로 쓰는 것만 손에 남긴다 |
| 내가 잘 쓰고 있는지 감이 안 옴 | 뭘 실제로 쓰는지 한 화면에 보여준다 |
| 한번 깔면 굳어서 실제 습관과 어긋남 | 안 쓰는 건 내리고 다시 쓰는 건 올리자고 제안한다 |

### pouch가 맡지 않는 자리

pouch는 에이전트를 더 똑똑하거나 빠르게 만들지는 않는다. 그리고 작업 *도중*에
생기는 아픔 — 뭘 할지는 아는데 중간에 막히거나, 내 의도와 다르게 흘러가서 다시
잡는 것 — 은 pouch가 직접 손대는 자리가 아니다. 그건 작업 실행을 지켜보는 다른
하네스가 더 잘하는 일이고, pouch는 그 앞단(내 환경과 기억)에 집중한다.

그래서 pouch가 없애는 건 성능 문제가 아니라 **성가심과 어긋남**이다: 매번 다시
설명하는 성가심, 안 쓰는 도구에 파묻히는 성가심, 세팅이 실제 나와 어긋나는 것.

## 왜 만드나

나를 위한 무기를 만드는 걸 좋아한다. 그런데 내가 날 세워 만든 게 옆 사람한테도 쓸모가 있고, 그 사람이 그걸 쓰고 고마워할 때 가장 큰 힘을 받는다.

`pouch`는 그 흐름을 그대로 설계에 담는다.

- **개인 하네스로 시작한다.** 누구를 위한 플랫폼이 아니라, 내 손에 맞는 나만의 주머니.
- **자연스럽게 퍼진다.** 해달들이 잘 때 떠내려가지 않으려 서로 손을 잡고 무리를 이루는 걸 *a raft of otters*라 부른다. 각자 띄운 주머니가 옆 사람과 묶여 한 뗏목(raft)이 되는 것 — 그게 이 프로젝트가 회사 전체로 가는 그림이다.

설계 1순위는 언제나 "내가 쓰고 싶은가"다. 전사 확산은 그 다음에 따라오는 결과지, 출발점이 아니다.

## 핵심 개념

### 🪨 나에게 맞춰진다

설치 마법사가 사용자의 환경·역할·취향을 묻고, 그에 맞는 도구/스킬 구성을 골라 주머니에 담는다. 선택지는 다양하되, 결과물은 **개인화된 한 벌**이다.

### 🌊 쓸수록 진화한다

한 번 깔고 끝나는 도구가 아니다. 쓰면서 닳고, 새 스킬이 붙고, 안 쓰는 건 떨어져 나간다. 주머니의 구성은 시간이 지날수록 사용자에게 더 정확히 맞아간다.

### 🦦 묶일 수 있다

개인 주머니로 출발하지만, 같은 구조를 공유하기에 옆 사람과 손쉽게 엮인다. 한 사람의 좋은 돌이 무리 전체로 번지는 경로가 처음부터 열려있다.

## 설계 원칙

> 아직 개발 초기라 아래 원칙은 "지키려는 방향"이지, "이미 다 된 것"이 아니다.

- **개인 우선.** 공용을 위해 개인을 희생시키지 않는다. 좋은 개인 도구가 좋은 공용 도구가 된다.
- **마법사 중심 셋업.** 사용자가 설정 파일을 외우는 게 아니라, 마법사가 물어보고 맞춰준다.
- **점진적 진화.** 한 번에 완성하지 않는다. 닳고 붙고 떨어지는 걸 전제로 구조를 짠다.
- **규칙은 코드로.** 문서로 적힌 약속보다, 잘못 쓰기 구조적으로 어려운 동작을 선호한다.

## 설치

**필요한 것:** Python 3.11+, [uv](https://docs.astral.sh/uv/), 그리고 에이전트 연결을 쓰려면 [Claude Code](https://claude.com/claude-code).

가장 간단한 방법 — 저장소를 받아 도구로 상주시킨다(어느 디렉토리에서든 `pouch`가 된다):

```bash
git clone https://github.com/HaeDalWang/Pouch.git
cd Pouch
uv tool install .
```

설치 확인:

```bash
pouch --version
pouch --help
```

> 개발용으로 clone 위치에서 바로 돌리려면 `uv run pouch …` 를 쓴다(상주 설치 없이).
> 나중에 걷어내려면 `uv tool uninstall pouch`.

새 버전으로 업그레이드(clone 디렉토리에서):

```bash
git pull && uv tool install . --force --reinstall
```

> `--force`만으로는 캐시된 wheel이 그대로 남아 버전이 안 올라갈 수 있다 — `--reinstall`까지 같이 줘야 확실하다.

## 빠른 시작 (end user)

주머니에 담을 것을 등록하고 → 나에게 맞춰 채우고 → 에이전트에 연결하고 → 쓸수록 진화시킨다.

```bash
# 0) 📚 쓸 만한 것을 "소스"에 가리켜 둔다 (아직 장부엔 안 들어감)
#    SKILL.md / .mcp.json / plugin 디렉토리를 자동 판별해 들인다.
#    import는 후보를 넓게 적어두기만 한다 — 실제로 쓰거나 install하면
#    그때 장부(catalog)로 "진입"해 pouch가 관리를 시작한다.
pouch catalog import <경로>     # 예: ~/.claude/plugins/cache/<플러그인>
pouch catalog list --sources    # 소스에 재워둔 것 확인
pouch catalog list              # 장부에 진입한 것 확인

# 1) 🪨 환경을 감지하고 나에게 맞춰 주머니를 채운다
#    역할·스택을 물어보고, 맞는 "시작 세트"(미리 꾸려진 한 벌)가 있으면
#    통째로 제안받는다 — 세트는 소스를 건너뛰고 곧장 장부로 진입한다.
#    완전 처음이라 소스·장부가 다 비어 있어도 세트로 시작할 수 있다.
pouch init                      # (pouch set list 로 세트 구경)

# 2) 🔌 에이전트(Claude Code)에 연결한다
#    - 새 세션마다 당신의 기억을 안고 시작하고 (SessionStart)
#    - 어떤 도구를 실제로 쓰는지 조용히 기억해둔다 (PostToolUse)
#    설정은 pouch가 대신 바꾸고, 되돌릴 수 있게 백업(.bak)을 남긴다.
pouch hook install

# 3) 주머니 상태 한 화면 — 뭐가 담겼고, 최근 뭘 썼고, 연결은 어떤지
pouch
```

이후 Claude Code를 쓰다 보면 사용 기록이 `~/.pouch/usage.jsonl`에 쌓인다. 며칠 뒤:

```bash
# 🌊 안 쓰는 건 내리고, 다시 쓰는 건 올리자고 제안받는다 (동의 없이는 아무것도 안 움직인다)
pouch evolve
```

`evolve`가 도구를 "내려도" 카탈로그와 당신의 개인화(경계·메모)는 남는다 — **떨어진다 ≠ 삭제된다.** 내렸던 도구를 다시 쓰면 evolve가 재부착을 제안하고, 주머니 밖에서 자주 쓰는 도구는 편입을 안내한다. 쌓아둔 경계는 전 과정에서 그대로 살아있다. 직접 올리려면 `pouch catalog install <id>`.

되돌리기: `pouch hook uninstall` 로 연결을 원상복구한다.

주머니를 통째로 지키고 싶으면 (잃었을 때 되찾기):

```bash
pouch backup                  # ~/.pouch를 아카이브 하나로 백업 (기본: ~/pouch-backups/)
pouch restore <아카이브 경로>  # 백업 시점으로 되돌림 — 덮기 전 현재 상태를 자동 스냅샷
```

> v0는 로컬 폴더 백업이다. 프로젝트별 `.pouch/`(민감정보)는 클라우드로 안 나가게
> 전역 `~/.pouch`만 담는다. S3·구글드라이브 목적지는 같은 코어 위에 얹을 예정.

기억을 직접 다루고 싶으면:

```bash
pouch memory list             # 담긴 기억 보기
pouch memory add -n <이름> -d <요약> -b <본문>
pouch memory recall <검색어>  # 키워드로 회상
```

## 로드맵

- [x] **메모리 레이어** — 쓸수록 쌓이는 개인 기억 + 에이전트 주입 (Phase 1)
- [x] **설치 마법사** — 환경·역할을 묻고 나에게 맞춰 담는다 `pouch init` (🪨 Phase 2)
- [x] **도구 카탈로그** — 무엇을 담을 수 있는지의 레지스트리 (owned/vendored/linked) (Phase 3)
- [x] **진화 엔진** — 사용을 추적해 안 쓰는 도구 정리 제안 `pouch evolve` (🌊 Phase 4)
- [x] **체감 루프** — 카탈로그 CLI·상태 화면·attach 제안 (Phase 4.6 ①~③)
- [x] **백업/복원** — `~/.pouch`를 잃었을 때 되찾기 `pouch backup`/`restore` (로컬 v0)
- [x] **사용 기록 위생** — 오래된 기록을 요약으로 접어 무한 성장 방지 (evolve 때 자동)
- [x] **담는 종류 6종 완성** — 스킬·명령·에이전트·규칙·훅·도구연결(mcp) 전부 (경계의 방향·출처 포함)
- [x] **시작 세트(배관)** — 미리 꾸려진 한 벌을 통째로 제안하는 길 `pouch set` (🪨 콜드 스타트)
- [x] **네이티브 메모리 대체** — Claude 자체 메모리를 pouch로 이관·청소 `pouch memory adopt`
- [x] **경계 1급화** — 자율성의 허용·확인·금지를 자기 명령으로 `pouch boundary`
- [x] **주머니 리포트** — 기간별로 뭘 쓰고 뭐가 닳는지 `pouch report`
- [x] **핵심 도구 인식** — 실사용에서 손에 맞은 도구를 배워 보호 (🪨 개인화 학습 첫 걸음)
- [x] **맥락 개인화 — 프로젝트별 주머니** — 프로젝트 사용을 보고, 카탈로그에 프로젝트 스코프를 주고, 리포트가 사용을 주머니 안/밖으로 갈라 편입을 안내 (로컬 전용, 레인 2)
- [x] **init 3축 재구성** — 역할·방향(자유 입력, 고정 6선택지 폐기)·말투·경계로 다시 묻고, 저장 직전 답을 2인칭 서사로 되읽어 확인 (`reflect` 되비춤)
- [x] **세트 내보내기** — 지금 표면(연장통)을 세트 파일로 굳힌다 `pouch set export` (🦦 내장 세트 1호·raft 공유의 문)
- [x] **기억 저마찰 축적** — 세션에서 배운 게 기억으로 흘러드는 경로: project·reference는 리뷰 스테이징(`--pending`), 민감 타입은 사용자 확인 (Phase 4.6 ④ 에이전트 배선)
- [x] **태그 승격(학습된 관심사)** — 손에 맞은 도구가 달고 온 토큰을 관심사로 승격, 추천이 실사용을 반영 (🪨 Phase 4.5, 실측 태그 0/201이라 설명 토큰으로)
- [ ] 주머니 공유 — 개인에서 팀으로 (🌊 raft, Phase 5)

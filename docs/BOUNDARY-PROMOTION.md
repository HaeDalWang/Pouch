# 🦦 경계를 1급으로 — 승격 옵션

> **이 문서는 포지셔닝 문서가 "가장 방어 가능한 해자"로 꼽은 경계(boundary)를
> 어떻게 "1급 개념"으로 올릴지 옵션을 펼치고 하나를 고르기 위한 것이다.** 코드 전
> 결정용 — 방향이 정해지면 여기 근거를 남긴다.
>
> 읽는 법은 [HOW-IT-WORKS.md](HOW-IT-WORKS.md)와 같다.

---

## 지금 경계는 이미 이만큼 한다

경계는 맨땅이 아니다. 이미 다음이 코드로 돌아간다:

| 있는 것 | 어디 |
|---|---|
| 타입(`boundary`) + 방향(`allow`/`ask`/`deny`) + 출처(`user`/`vendored:<도구>`) | `memory/model.py` |
| 세션 주입 시 **맨 위에 본문까지**, `[DENY]` 라벨이 산문보다 우선 | `memory/context.py` |
| 도구 내릴 때 방향으로 가르기(허용은 함께 강등, 금지·확인은 잔존) | `catalog/boundary.py` |
| 허용 누수 방지 = **저장 위치**(프로젝트 경계는 그 repo에서만 읽힘) | scope 격리 |
| **강제하지 않음**(차단기 안 닮) — "알고 존중하게" 하는 층 | 의도된 결정 |

## 그런데 "1급"은 아니다 — 세 구멍

1. **짓는 입구가 묻혀 있다.** 경계를 걸려면 `pouch memory add -t boundary --direction deny`
   — 메모리의 곁다리다. 경계 자신의 동사가 없다.
2. **한눈에 보는 곳이 없다.** `pouch memory list`에 다른 기억과 섞인다. *"여기서 내가
   못 하는 게 뭐지?"* 를 한 화면에 못 본다.
3. **온보딩이 없다.** `init`이 경계를 안 묻는다. 콜드 스타트에 경계는 0개로 시작.

## 옵션

| 옵션 | 무엇 | 평가 |
|---|---|---|
| **A. 전용 명령 표면** | `pouch boundary add/list/remove` — 저장 모델은 그대로(여전히 memory), **CLI 표면만 1급**. list는 방향·스코프별 그룹 | 낮은 위험 · 높은 UX · 모델/철학 안 건드림 |
| **B. 온보딩** | 흔한 경계 템플릿(force-push 금지, prod 확인 등)을 init/wizard가 **제안** | 콜드 스타트 가치↑. "지어낸 세트 안 담는다" 철학과 닿아 신중히(기본값 아닌 "물어보기") |
| **C. 별도 저장/모델 분리** | 경계를 memory store 밖 독립 저장소로 | 큰 리팩터 · 이득 의문(이미 scope 격리됨) · **비추** |
| **D. 강제(PreToolUse 차단)** | 훅으로 DENY 행위를 실제 차단 | ❌ **철학이 명시적으로 거부** — "pouch가 막아주니 괜찮겠지"라는 거짓 안전감이 차단 없음보다 위험. **후보 아님** |

## 추천: A(지금) → B(다음)

- **A**가 경계를 *"자기 동사를 가진 것"*으로 만들어 1급화하면서, 저장 모델도 비강제
  철학도 안 건드린다. 가장 값싸고 방향이 맞는 첫 걸음.
- **B**는 콜드 스타트 가치가 크지만 템플릿 큐레이션이 "남이 만든 정답 세트" 철학과
  닿아 신중해야 한다 — A 다음에, 제안(물어보기) 형태로.
- **C**(과설계)·**D**(철학 위반)는 제외.
- ✅ **D 제외는 오너 확인됨(배승도, 2026-07-16).** 오너의 말: *"차단기를 만들었다가
  애초에 유저가 이루려는 목적에 도달도 못 할 것 같아서 안 만드는 그 판단은 지금도
  그대로가 맞다."* — 기존 근거(거짓 안전감)와 나란히 서는 오너의 두 번째 근거:
  잘못 막는 차단기는 사용자의 목적 달성 자체를 방해한다.

## A를 고르면 — 제안하는 구체 형태

```
pouch boundary list              # 모든 경계를 스코프·방향별로 한 화면
pouch boundary add -n <name> -d <summary> -b <body> --direction <allow|ask|deny> [-s <scope>]
pouch boundary remove <name>     # = 경계 forget
```

- **`list`**: 스코프별 그룹, 각 줄 `[DENY] name — summary` + 출처 표시(🔒 내가 건 것 /
  📦 도구가 딸고 온 것). "이 프로젝트에서 내 자율 범위"를 한눈에.
- **`add`**: `--direction`을 **필수**로 요구한다 — 방향 없는 경계가 gate에서 "잘 모름
  (잔존)"으로 처리되는 애매함을, 짓는 입구에서부터 막는다(`memory add`의 boundary
  경로는 방향 선택). 출처는 언제나 `user`(CLI로 vendored 참칭 불가, 기존 규칙 유지).
- **`remove`**: 경계는 `pouch memory forget`으로도 지워지지만, 전용 동사를 준다.
- **하위호환**: `pouch memory add -t boundary`는 그대로 둔다(안 깨뜨림). `boundary add`가
  승격된 정식 입구.

## 구현됨 · 오너 확인됨 — A의 형태

> ✅ **A는 오너 확인됨(배승도, 2026-07-16).** 오너의 말: *"규칙이라는 문을 보기
> 쉽게 간판을 달고 찾을 수 있게 한 것뿐!"* — 세기·강제와 무관하다는 것("규칙의
> 힘은 그대로")을 이해한 위에서의 승인. 이 걸림에서 GLOSSARY "1급" 항목이 태어났다.

`pouch/boundary/` 패키지 + `cli.py` 등록으로 최상위 `pouch boundary` 명령을 냈다.

1. 서브 vs 최상위 — 제안: **최상위 `pouch boundary`**(1급화 취지). memory 곁다리가 아니라 자기 동사.
2. list에 archived/pending 표시 — 제안: **indexed만**. 강등된 경계는 효력이 없어 "지금 내 범위"에서 뺀다.
3. edit 추가 — 제안: v0는 **remove + add**로 갈음(불변 값 객체와 결이 같다).

`add --direction` **필수**로 방향불명 애매함을 입구에서 차단, 출처는 언제나 `user`(참칭 불가),
저장은 여전히 memory(모델 안 바꿈), `memory add -t boundary`도 하위호환 유지. **비강제 철학
그대로** — list/add/remove는 표면일 뿐, 차단기는 없다.

## B(온보딩) — 완료

`init`이 흔한 안전 경계(force-push·prod·secrets)를 **제안**한다(`boundary/templates.py` +
`_maybe_offer_boundaries`). 철학을 지키는 두 장치:

- **안전 경계만, near-universal한 것만** — 생산성 큐레이션은 안 담는다.
- **고른 것만 담긴다** — 기본값 아닌 checkbox 제안. `--yes`(비대화형)에선 아예
  건너뛴다(경계는 감지된 사실이 아니라 선호라 강요 안 함). 이미 걸린 건 후보에서 뺀다.

진짜 경계는 실사용에서 사람이 `pouch boundary add`로 건다 — 템플릿은 씨앗일 뿐.

> ✅ **B는 오너 확인됨(배승도, 2026-07-16).** "지금도 강제가 아니라 골라야만
> 담긴다"를 확인한 뒤 **유지**를 선택. 시작 세트에 경계를 담자는 아이디어도
> 나왔으나 별개 설계라 나중으로(예약 아님). A(전용 명령)도 같은 날 확인됨(위 참조).

# 🦦 pouch는 어떻게 도는가

> **이 문서는 pouch의 오너를 위한 문서다.** 코드를 안 열어도, ROADMAP의 압축된 용어를
> 몰라도, pouch가 지금 실제로 어떻게 동작하는지 처음부터 끝까지 따라갈 수 있게 쓴다.
>
> **읽는 법:** 읽다가 막히는 단어가 나오면 그냥 넘어가지 말고 표시해둔다.
> 그 단어들이 [GLOSSARY.md](GLOSSARY.md)(용어 사전)의 항목이 된다.
> 쉬운 말을 먼저 쓰고, 프로젝트에서 쓰는 용어는 괄호로 병기한다.

---

## pouch의 자리 한 줄

pouch는 에이전트를 쓰기 *전·후*의 **내 환경과 기억**을 맡는다 — 매번 나를 다시
설명하는 성가심, 안 쓰는 도구 산더미, 실제 나와 어긋난 세팅을 없앤다. *작업
도중* 에이전트가 막히거나 의도를 벗어나는 것은 pouch가 손대는 자리가 아니다
(그건 실행을 지켜보는 다른 하네스의 몫). 이 아래는 그 "전·후"가 어떻게 도는지의
자세한 이야기다. → 쓰는 사람 관점의 이득은 [README](../README.md)의 "이걸 쓰면
뭐가 달라지나".

## 한 장 요약

pouch가 하는 일은 동사 다섯 개다. 이 다섯 개가 한 바퀴 도는 것이 pouch의 전부다.

```text
        ①가리킨다          ②채운다            ③연결한다
   쓸 만한 걸 소스에 →  나에게 맞는 걸 골라 →  Claude Code에
   적어둔다              설치 (init)          꽂는다 (hook install)
   (catalog import)                              │
        ⑤제안받는다                            ④기록된다
   안 쓰는 건 내리자,   ←──────────────   세션에서 뭘 썼는지
   다시 쓰는 건 올리자                       조용히 쌓인다
   (evolve)                                 (usage.jsonl)
```

**한 곳만 미리 짚어둔다 — "가리킨다"와 "쓸 수 있다" 사이엔 문턱이 하나 있다.**
①에서 import한 것은 곧바로 장부(창고)에 들어가지 않는다. 일단 **소스**라는
대기 자리에 재워두고, 네가 **실제로 쓰거나 손수 올릴 때만** 장부로 "진입"한다.
왜 이렇게 바꿨고, 진입하면 나에게 뭐가 달라지는지는 ①에서 자세히 다룬다 —
이 문서에서 가장 최근에 바뀐, 그리고 가장 오해가 많은 대목이다.

그리고 이 바퀴와 나란히, **기억**이 같은 모양의 생애주기를 돈다
(쌓인다 → 매 세션 주입된다 → 낡은 것은 내려간다). 뒤에서 따로 다룬다.

### 디스크 지도 — 내 컴퓨터 어디에 뭐가 있나

| 위치 | 무엇 | 한 줄 설명 |
| --- | --- | --- |
| `~/.pouch/sources/` | 소스 대기 자리 | import했지만 **아직 안 써서 장부엔 안 든 것**. 백과사전에 있으나 내 노트엔 없는 페이지 |
| `~/.pouch/catalog/` | 도구 장부(카탈로그) | 실제로 **진입한** 도구의 목록. pouch가 "네 도구"로 관리하는 것. 파일 하나 = 도구 하나 |
| `~/.pouch/memory/` | 전역 기억 | 어느 프로젝트에서든 통하는 나에 대한 기억 |
| `<프로젝트>/.pouch/memory/` | 프로젝트 기억 | 그 저장소 안에서만 읽히는 기억 (민감한 것은 여기로) |
| `~/.pouch/repos/<이름>/` | 등록한 저장소 | `pouch repo add`로 물어둔 바깥 주소의 사본 (helm repo처럼). 장부 파일 없음 — 클론과 git remote가 유일한 진실 |
| `~/.pouch/repo-index/<이름>/` | 저장소 색인 | 그 사본을 훑어 만든 도구 목록(파생물 — 재색인 때 지우고 다시 만듦). 저장소 이름은 자리가 답한다 |
| `~/.pouch/usage.jsonl` | 사용 기록 | "언제 어떤 도구를 썼다" 한 줄씩 덧붙는 로그 |
| `~/.pouch/state.json` | 설치 상태 | 지금 어떤 도구가 올라가 있는지 |
| `~/.claude/skills/`, `agents/`, `commands/`, `.mcp.json` | **표면** | Claude Code가 실제로 읽어가는 자리. 여기 있어야 에이전트가 쓴다. **종류마다 서랍이 다르다** |
| `~/.claude/settings.json` | Claude 설정 | pouch가 hook 두 개를 여기에 등록한다 (백업 `.bak` 동반) |

경로 결정 로직은 전부 한 파일에 모여 있다: [src/pouch/paths.py](../src/pouch/paths.py)

**"표면"이라는 말** — pouch 문서 곳곳에 나오는 이 단어는 "에이전트가 실제로 도구를
꺼내 쓰는 자리"라는 뜻이다. 넓은 작업 환경(홈디렉토리 같은 것)이 아니라, 공사현장으로
치면 **"그날 연장통에 꺼내놓은 연장"** 같은 특정 자리다 — 구체적으로 `~/.claude/skills/`
폴더와 `.mcp.json` 파일. 장부(카탈로그)가 창고 재고라면 표면은 연장통이다. 창고에
있어도 연장통에 없으면 그날 못 쓴다. "설치한다 = 연장통에 올린다", "내린다 =
연장통에서 뺀다(창고엔 그대로)"로 읽으면 된다. → 더 자세히는 [GLOSSARY.md](GLOSSARY.md).

**연장통에는 서랍이 나뉘어 있다.** 에이전트가 종류마다 다른 자리를 읽기 때문이다.
pouch가 분류를 새로 발명하지 않고, 에이전트가 이미 나눠둔 서랍을 그대로 쓴다:

| 종류 | 올라가는 자리 | 모양 |
| --- | --- | --- |
| 스킬 | `~/.claude/skills/<이름>/SKILL.md` | 폴더를 만들고 그 안에 |
| 에이전트 | `~/.claude/agents/<이름>.md` | 파일 하나로 평평하게 |
| 명령 | `~/.claude/commands/<이름>.md` | 파일 하나로 평평하게 |
| 규칙 | `~/.claude/rules/<묶음>/<이름>.md` | 원래 있던 폴더를 되살려서 |
| 훅 | `~/.claude/settings.json` | 설정에 등록 |
| MCP | `.mcp.json` | 설정에 등록 |

**규칙만 이름이 접혀 있다가 펴진다.** 규칙 파일은 `rules/python/coding-style.md`처럼
폴더 한 겹 안에 사는데, `coding-style.md`라는 이름이 python·golang·common 여러 묶음에
겹친다. 창고 장부는 평평해서(파일 하나 = 항목 하나) 담을 때 `python__coding-style`처럼
폴더를 `__`로 **접어** 넣고, 표면에 올릴 때 다시 폴더로 **편다**. 절차가 한 단계 늘지만
내려놓는 자리가 주워온 자리와 같아진다 — 어긋나면 "되돌려놨다"가 거짓말이 된다.
접기·펴기는 한 파일에 같이 산다([src/pouch/catalog/docid.py](../src/pouch/catalog/docid.py)):
따로 떨어져 있으면 한쪽만 고쳐져 왕복이 깨진다.

규칙은 머리말(`---` 로 둘러싼 부분) 없이 본문만 쓴다. 하네스가 규칙 파일을 평문으로
통째로 읽어 지침에 싣기 때문에, 머리말을 얹으면 그 글자가 지침 안에 그대로 섞인다.

**서랍이 없는 종류는 여전히 거절한다.** 아무 데나 놓느니 "올릴 자리가 정해져 있지
않습니다"라고 정직하게 말한다. Claude 밖(Codex·Kiro)에는 규칙에 해당하는 자리가
확인된 바 없어 pouch가 지어내지 않는다 — 표면은 지금도 Claude만 대상이다.

> 2026-07-21 이전에는 훅·MCP를 뺀 **전부가 스킬 서랍으로** 갔다. 에이전트를 올려도
> 스킬인 척 잘못 놓여서, 에이전트로는 살지 않았다. 창고가 9개일 땐 안 드러났는데
> 훑기로 451개가 들어오면서 보였다.
>
> 그때 **내리는 쪽은 같이 안 고쳐졌다**(2026-07-22 발견·수리). 올리기는 서랍별로
> 갔는데 내리기는 계속 스킬 서랍만 뒤져서, 에이전트·명령을 내려도 파일이 표면에
> 남아 계속 읽혔다 — "내렸습니다"가 거짓말이던 자리다. 이제 올릴 때 쓴 자리 계산을
> 내릴 때 그대로 되쓴다. 규칙은 폴더째가 아니라 **파일 하나만** 지운다(같은 묶음의
> 이웃 규칙이 말없이 쓸려가지 않게).

## 등장인물: 장부와 기록은 분리되어 있다

pouch 구조에서 가장 중요한 설계 하나만 꼽으면 이것이다:

> **도구의 "정의"(장부)와 도구의 "이력"(사용 기록·설치 상태)은 다른 파일에 산다.**

- 장부(`catalog/`)는 깨끗하게 유지된다 — 도구가 무엇인지만 적혀 있다.
- 이력(`usage.jsonl`, `state.json`)은 옆에 붙은 별도 파일이다.
  프로젝트 용어로 **사이드카(sidecar)** — 오토바이 옆에 붙는 보조석처럼,
  본체를 더럽히지 않고 옆에서 기록만 쌓는 파일이라는 뜻이다.

이 분리 덕분에 "도구를 내려도 기록은 남는다", "기록을 지워도 장부는 무사하다"가
구조적으로 보장된다. 뒤에 나오는 **"떨어진다 ≠ 삭제된다"** 원칙의 물리적 토대다.

## ① 가리킨다 — `pouch catalog import <경로>`

> **이 섹션은 최근에 크게 바뀌었다.** 전에는 import가 곧바로 장부에 등록했다.
> 지금은 **소스 대기 자리에 재워두기만** 한다. 왜 바꿨는지, 그래서 나에게 뭐가
> 달라지는지가 이 섹션의 핵심이다.

경로를 주면 pouch가 그것이 무엇인지 스스로 판별해서 들여온다:

- `SKILL.md` 파일 → 스킬 하나로
- `.mcp.json` → 외부 서버(MCP) 연결 정보로
- `hooks.json` → 훅("이 사건이 나면 이 명령을 실행하라"는 배선) 낱개로.
  훅은 **내 컴퓨터에서 명령을 실제로 실행**하므로 설치할 때 실행될 명령
  원문을 반드시 보여주고 동의를 받는다 — `--yes`로 물음은 건너뛰어도
  원문 출력은 항상 남는다
- 플러그인 디렉토리 → **통째로가 아니라 낱개로 쪼개서** (플러그인은 스킬 수십 개
  묶음이라, 묶음째 두면 "이 중 뭘 쓰는지"를 알 수 없게 되기 때문)

### 경로를 손으로 안 찍어도 된다 — `pouch catalog sweep`

여기엔 오래된 구멍이 하나 있었다. pouch는 언어 런타임(python·node)은 알아서
감지하면서, **이미 깔려 있는 도구는 하나도 찾아보지 않았다.** 사람이 경로를 손수
찍어줘야만 들어왔다. 그래서 스킬이 160개 깔린 컴퓨터에서도 주머니는 텅 빈 채로
시작했다(2026-07-21 실측 — 장부에 9개뿐이었다).

`sweep`이 그 구멍을 막는다. **에이전트마다 도구를 어디 두는지 알고, 그 자리를
한 번 훑어서 소스에 재운다.**

| 에이전트 | 훑는 자리 |
| --- | --- |
| Claude Code | `skills/` · `agents/` · `commands/` · `rules/` · `plugins/cache/` · `.mcp.json` |
| Codex | `~/.codex/skills/`, `~/.codex/hooks.json` |
| Kiro | `~/.kiro/skills/` |

`agents/`·`commands/`·`rules/`는 폴더 안에 `.md`가 평평하게 놓인 모양이라, **파일만
봐서는 무엇인지 알 수 없다** — `agents/`에 있으면 에이전트, `commands/`에 있으면
명령이라고 **자리가 답한다**. 규칙만 `rules/<분류>/<이름>.md`로 한 겹 더 들어가고,
같은 파일명이 여러 분류에 겹치므로 `분류__이름`으로 구분한다.

**이름이 겹치면 덮지 않고 거절한다.** 장부는 `<이름>.md` 한 자리라 종류가 달라도
이름이 같으면 자리를 다툰다. 실측에서 `agent-sort`·`context-budget`·`rules-distill`이
스킬과 명령으로 둘 다 있었는데, 나중에 온 것이 앞의 것을 소리 없이 지웠다. 지금은
먼저 담긴 것을 지키고 이유를 알린다 — **조용한 손실보다 시끄러운 거절이 낫다.**

**훑기는 소스까지만 간다.** 장부에 넣지도, 연장통에 올리지도 않는다 — 그건 그대로
실사용이 정한다(바로 아래 절의 규칙이 그대로 산다). 그래서 훑기는 안전하고,
`pouch init`이 물어보지 않고 자동으로 한 번 돌린다.

> 왜 자동인가: 이 도구를 모르는 사람은 "경로를 찍어야 한다"는 걸 영영 모른 채
> 빈 주머니로 산다. 찾아서 **재우는** 것까지는 되돌릴 게 없으니 자동으로 하고,
> 되돌릴 것이 생기는 지점(장부 진입·연장통)부터 사람이 정한다.

두 번 훑어도 두 번 담기지 않는다(이미 아는 것은 그대로 둔다). 깨진 스킬 하나가
나머지를 막지도 않는다 — 건너뛴 것은 이유와 함께 보고된다.

**이미 pouch를 쓰던 사람은 어떻게 되나.** `init`은 처음 한 번만 도니까, 쓰던
사람에게는 훑기가 저절로 돌지 않는다. 그래서 `pouch` 첫 화면이 **딱 한 번**
알려준다 — 무엇을 하면 무엇을 얻는지까지 한 줄에 못박아서:

```text
담긴 것  카탈로그 비어 있음
     ! 이미 깔린 도구를 아직 안 찾아봤습니다 — pouch catalog sweep 을 해야 대기실에 들어갑니다
```

한 번 훑고 나면 이 줄은 사라지고 "소스 N개 대기"로 바뀐다. 실행은 여전히 사람이
정한다 — pouch는 알려주기만 한다.

새 에이전트가 생기면 **그 에이전트의 어댑터에 "도구를 어디 두나"만 채우면**
훑기에 자동으로 끼어든다. 훑기 쪽 코드에는 에이전트 이름이 하나도 박혀 있지 않다.

### 바깥 주소를 물어두기 — `pouch repo` (Phase 4.8 ①)

훑기(sweep)가 "내 컴퓨터에 이미 깔린 것"을 줍는다면, 저장소 등록은 **바깥의
주소를 물어두는** 문이다. helm을 아는 사람에겐 그대로다: `helm repo add`처럼
주소를 등록해두면, 그 안의 도구들이 나중에 검색·추천·설치의 재료가 된다.

```sh
pouch repo add <이름> <git주소>   # 등록 (다시 치면 갱신 — 멱등)
pouch repo list                   # 등록 목록 (처음엔 빈손이 정상)
pouch repo remove <이름>          # 등록 해제 (주소만 알면 언제든 다시 add)
```

알아둘 것 셋:

- **등록 = 신뢰 표명.** 어느 주소를 물릴지는 사람이 고른다. pouch가 기본 저장소를
  미리 물려두지 않는 이유 — 미리 물려두면 "pouch가 고른 정답"이 되기 때문이다
  (내장 세트를 철회했던 것과 같은 교훈).
- **등록해도 설치·실행은 없다.** 사본을 받고 **색인까지만** 한다 — add가 사본을
  훑어(로컬 훑기와 같은 기계, 아는 배치만: skills/·agents/·commands/·rules/·
  .mcp.json) "이 저장소에 뭐가 있나" 목록을 만들고 몇 개 알아봤는지 말해준다.
  모르는 구조는 안 잡히고, 깨진 조각은 이유와 함께 건너뛴다. 색인은 사본의
  파생물이라 재등록(갱신) 때마다 지우고 다시 만든다 — 바깥에서 사라진 도구가
  유령으로 안 남는다. 같은 이름을 다른 주소로 덮으려 하면 거부한다.
- **색인은 검색되고, 추천에 합류하고, 설치까지 이어진다.** `pouch repo search
  <말>`이 등록한 저장소 전부의 색인을 뒤진다(`helm search repo`의 pouch판) —
  빈손이어도 쓸 수 있는 입구다. '이거 써봐'가 저장소 색인을 후보 풀의 다음 겹으로
  보고(아래 evolve 절 참조), 마음에 들면 `pouch catalog install <저장소>/<도구>`로
  표면까지 올린다. 진입하면 카탈로그 안 정체는 맨 이름이 된다(출처는 upstream이
  말한다) — 이후는 여느 도구와 똑같이 서랍·동의 관문·내리기/되올리기를 탄다.
  같은 이름이 딴 데서 이미 와 있으면 조용히 덮지 않고 거부한다.
  세트가 저장소를 참조하는 것(⑤)은 다음 조각.

### 담는다가 아니라 "가리킨다" — import는 장부에 안 넣는다

여기가 가장 큰 변화다. `import`가 하는 일은 이제 **소스(`~/.pouch/sources/`)에
적어두는 것**뿐이다. 장부(`catalog/`)에는 아직 아무것도 안 들어간다.

> 비유: import는 **백과사전에 그 항목이 있다고 표시**하는 것이지, **내 노트에
> 옮겨 적는** 게 아니다. 노트(장부)에는 내가 실제로 펼쳐 본 페이지만 올라간다.

**왜 이렇게 바꿨나 (관문 "다" 정책).** 전에는 플러그인 하나를 import하면 그 안의
스킬 수백 개가 몽땅 장부로 직행했다. 실제로 쓰는 건 두세 개인데, 장부가 안 쓰는
도구로 뒤덮여 "내가 진짜 뭘 쓰는지"의 신호가 묻혔다. 그래서 규칙을 뒤집었다 —
**import는 후보를 넓게 가리키기만 하고, 무엇이 장부에 들어갈지는 실사용이 정한다.**

### 소스 → 장부 "진입"은 언제 일어나나

소스에 재워둔 것이 장부로 올라오는 것을 **진입**이라 부른다. 진입 트리거는 셋:

| 트리거 | 명령 | 뜻 |
| --- | --- | --- |
| **실제로 썼다** | (자동) `pouch evolve` | 세션에서 그 도구를 한 번이라도 쓰면, evolve가 소스에서 찾아 장부로 올린다. 가장 중요한 경로 |
| **손수 골라 올렸다** | `pouch catalog install <id>` | 아직 안 썼어도 "이건 쓸 거야"라고 명시하면 진입. 일부러 고른 것도 실사용과 같은 근거 |
| **세트가 적용됐다** | `pouch set apply <이름>` | 시작 세트가 가리킨 것을 통째로 진입 |

문턱은 **딱 한 번**이다. 우연히 스친 게 아니라 "내가 그 번들을 일부러
import했다"에 더해 "실제로 썼다"까지 겹치면, 그건 우연이 아니라서 3회씩 볼
필요가 없다.

### 진입하면 나에게 뭐가 달라지나 — 이게 핵심이다

많은 사람이 "import하면 pouch가 뭘 해주나?"에서 헷갈린다. 답은 **소스에 있는
동안은 아무것도 안 해준다.** pouch는 소스를 **일부러 못 본 척한다.** 장부에
진입해야 비로소 pouch의 관리가 시작된다:

| pouch가 하는 일 | 📚 소스에 있을 때 | 📦 장부에 진입한 뒤 |
| --- | --- | --- |
| `pouch catalog list`에 뜨나 | ❌ (안 뜸 — `--sources`로만 보임) | ✅ 뜬다 |
| `pouch`(상태 화면)가 세나 | ❌ 무시 | ✅ "카탈로그 N개"에 포함 |
| init 추천 후보가 되나 | ❌ 안 골림 | ✅ 역할·스택에 맞으면 추천 |
| evolve가 "안 쓰네, 내리자" 챙기나 | ❌ 관심 없음 | ✅ 사용 신호를 근거로 제안 |
| evolve가 "다시 쓰네, 올리자" 챙기나 | ❌ | ✅ |
| 표면(연장통)에 올릴 수 있나 | ❌ (먼저 진입해야) | ✅ `install`로 올림 |

**한 줄 요약:** 소스에 있는 것은 pouch에게 "존재는 아는데 관심 없는 것"이고,
장부에 진입한 것은 "내가 관리하는 네 도구"다. **진입 = pouch가 그 도구의 생애주기
(추천·사용추적·오르내림 제안)를 챙기기 시작하는 순간.** 그래서 import만 하고
"왜 아무 일도 안 일어나지?" 싶으면, 그건 정상이다 — 한 번 써보거나 `install`하면
그때 장부로 들어와 pouch의 레이더에 잡힌다.

지금 소스에 뭐가 재워져 있는지: `pouch catalog list --sources`

### 옛날에 담긴 잉여를 되돌리기 — `pouch catalog migrate`

관문 "다" 이전에 import한 것들은 옛 규칙대로 장부에 직행해 있다(실측 약 200개).
이 일회성 명령이 그중 **한 번도 안 쓴 것을 소스로 되돌린다** — 실제로 쓴 것만
장부에 남긴다. evolve 자동 루프엔 없다: 장부 파일을 지우는 파괴적 동작이라
사용자가 의도적으로 부르고, **적용 전 자동 백업**을 뜬다(되돌리려면 `pouch restore`).

- `pouch catalog migrate --dry-run` — 뭐가 내려갈지 목록만
- `pouch catalog migrate --yes` — 백업 뜨고 실행
- 강등 계산·적용: [src/pouch/evolution/reconcile.py](../src/pouch/evolution/reconcile.py) ·
  [src/pouch/catalog/demote.py](../src/pouch/catalog/demote.py) ·
  [src/pouch/evolution/orchestrate.py](../src/pouch/evolution/orchestrate.py) (`plan_migrate`/`migrate`)

### 들여올 때 도구마다 "관계"가 정해진다 (ownership)

들여오는 모든 도구는 pouch와의 관계가 셋 중 하나로 정해진다(소스에 담길 때
정해지고, 장부로 진입해도 그대로 유지된다). 판별 기준은 단 하나 —
**"따라가야 할 원본(upstream)이 밖에 있는가?"**

| 관계 | 뜻 | 비유 |
| --- | --- | --- |
| `owned` | 원본이 없다. 내용까지 내 것, 마음대로 고침 | 내가 쓴 노트 |
| `vendored` | 원본이 밖에 있다. 내용은 원본을 따라가고, 내 메모는 **덧붙임 종이(overlay)**에 분리 | 도서관 책 + 포스트잇 |
| `linked` | 실행 자체가 바깥에서 일어난다. 연결 방법만 등록 | 전화번호부의 연락처 |

도서관 책(vendored)에 직접 밑줄을 그으면 새 판이 나올 때 밑줄이 사라진다.
그래서 내 개인화(태그·경계·메모)는 반드시 포스트잇(overlay)에 쓴다 —
원본이 갱신(`pouch catalog sync`)돼도 포스트잇은 살아남는다.

- 판별·분해: [src/pouch/catalog/importer.py](../src/pouch/catalog/importer.py)
- 도구 한 건의 스키마: [src/pouch/catalog/model.py](../src/pouch/catalog/model.py)
- 원본 따라 갱신: [src/pouch/catalog/sync.py](../src/pouch/catalog/sync.py) —
  원본 경로가 버전 업데이트로 사라지면 형제 버전을 찾아 자동 이사(rehome)시키고,
  엉뚱한 스킬로 바뀌지 않았는지 이름으로 검증한다: [src/pouch/catalog/rehome.py](../src/pouch/catalog/rehome.py)

## ② 채운다 — `pouch init`

설치 마법사. 네 가지를 순서대로 한다:

1. **감지** — OS, shell, 언어 런타임, Claude 설치 여부를 훑는다
   ([src/pouch/init/detect.py](../src/pouch/init/detect.py))
2. **질문** — 세 가지를 묻는다: **역할·방향**(자유 입력 — "지금 뭘 하고
   앞으로 뭘 하고 싶은지". 감지가 자신있게 틀리는 자리라 고정 선택지 대신
   사람 말로 받는다), **주력 스택**(감지 결과를 선택지로), **말투**(친절/건조/중간).
   끝에 **경계**("절대 안 했으면 하는 것")를 자유 입력으로 묻되 비우면 건너뛴다.
   이 답들을 저장 직전 2인칭 서사로 **되읽어 확인받는다**(판정이 아니라 "이게
   날 아네"의 recognition — 자세함은 질문 수가 아니라 이 되비춤에서 나온다)
3. **저장** — 답을 기억으로 저장한다. 역할·스택·말투·환경은 `user` 타입,
   경계만 `boundary` 타입(방향 deny·사람 출처)으로 갈린다
   ([src/pouch/init/profile.py](../src/pouch/init/profile.py))
4. **도구통 훑기** — 에이전트마다 도구를 두는 자리를 한 번 훑어 소스에 재운다
   (①의 `sweep`. 묻지 않고 자동 — 소스까지만 가므로 되돌릴 게 없다)
5. **시작 세트 제안** — 역할·스택에서 관심 키워드를 뽑아, 맞는 시작 세트가
   있으면 통째로 제안한다 (아래 "시작 세트" 참조). 이어서 아직 연결 안 됐으면
   ③의 hook 연결도 제안한다 ([src/pouch/init/commands.py](../src/pouch/init/commands.py))

**알아둘 규칙 하나:** 세트를 고르는 근거는 둘이다 — **내가 답한 역할·스택**(의도)과,
**실사용에서 배운 관심사**(손에 맞은 핵심 도구들이 달고 온 설명 단어, 상태 화면의
"배운 관심사"가 그것). 깔려만 있는 환경(사실)은 여전히 근거가 아니다: 컴퓨터에
ruby가 깔려 있다고 ruby 세트를 권하지 않는다 — **내가 답했거나 실제로 써온 것만**
기준이다. 처음 쓰는 사람은 사용 이력이 없으니 답만으로 고른다.

**낱개 도구 추천은 아직 배선 전이다(솔직한 구멍).** "역할·스택 키워드 → 장부
도구 태그 매칭"의 계산 자체는 [recommend.py](../src/pouch/catalog/recommend.py)에
`recommend()`로 짜여 있지만, 지금 init은 이걸 부르지 않는다 — 세트 제안까지만
한다. 다만 후보가 없던 사정은 훑기(`sweep`)가 생기면서 달라졌다 — 이제 처음 켠
기계에도 소스에는 재워둔 것이 쌓인다(장부는 여전히 실사용이 채운다). 낱개 추천을
붙일 바탕은 생긴 셈이고, 그걸 실제로 잇는 것은 아직 안 한 일이다.

### 완전 처음이라면 — 시작 세트

처음 쓰는 사람은 소스도 장부도 다 비어 있어서, 추천이 고를 후보 자체가 없다.
그래서 init은 역할·스택 답을 받은 뒤 **맞는 시작 세트가 있으면 그걸 통째로
제안**한다. 좋다고 하면 세트가 적어둔 출처에서 가져와 채우고, 골라둔 것만
연장통(표면)에 올린다.

**세트는 "일부러 고른 것"이라 소스를 건너뛰고 곧장 장부로 진입한다.** ①의
문턱(소스 대기)은 "전량 자동 등록"을 막으려는 것인데, 세트는 사람이 고른
선곡표라 그 우려가 없다 — install이 진입 트리거인 것과 같은 이치다.

**단, 내장 세트는 지금 의도적으로 비어 있다.** 첫 시도로 "DevOps면 이걸 쓰겠지"
하고 손으로 고른 세트를 넣었다가, 실제 사용과 다르다는 오너 검증으로 철회했다 —
지어낸 세트는 pouch가 반대하는 "남이 만든 정답 세트"와 같기 때문이다. 내장 세트는
**실제로 쓰면서 손에 맞게 닳은 주머니를 굳힌 것만** 들어올 수 있다. 굳히는 명령은
이제 있다 — `pouch set export <이름>`이 지금 연장통(표면)을 세트 파일로 굳힌다
(`~/.pouch/sets/`). 알아둘 것: 명령은 문만 열지 심사는 안 한다 — 갓 올린 것도
담기니, 내장 후보로 굳힐지는 여전히 사람이 판단한다.

**직접 만든(owned) 도구도 이제 세트에 담긴다 — 본문을 통째로 실어서(owned
임베드, 2026-07-18 오너 락: 인라인).** 남이 만든 도구는 "어디서 가져올지"
주소만 적으면 되지만, 직접 만든 도구는 가져올 원본이 없다. 그래서 세트 파일
안에 본문(실물)을 그대로 끼워 넣는다 — 노래 목록에 비유하면 남의 노래는 "어느
앨범 몇 번 트랙", 내 자작곡은 악보를 통째로 끼우는 격. 받는 사람은 세트 파일
하나만 받으면 그 안의 자작 도구까지 자기 주머니에 owned로 생긴다. 받는 쪽
울타리 둘: ① 임베드 이름에 `../` 같은 탈출 꼴이 있으면 거부(남이 준 파일이
내 폴더 밖에 쓰는 것 방지) ② 같은 이름이 이미 내 주머니에 있으면 덮지 않는다
(내가 깎아온 본문 보호). 지금은 글 한 장짜리(SKILL.md) 도구만 — 딸린 스크립트가
여럿인 도구는 본문만 실리므로, 그런 건 건너뛰지 않되 반쪽임을 사람이 알아야
한다(여러 파일 동봉은 예약된 숙제).

**저장소에서 온 도구는 주소로 가리켜 담긴다(Phase 4.8 ⑤).** `pouch repo add`로
등록한 저장소에서 설치한 도구는, 세트에 실릴 때 본문도 경로도 아니라 **저장소
참조**(이름·주소·도구 이름들)로 굳는다 — 받는 사람 컴퓨터엔 내 클론 경로가
없으니까. 이게 원격 MCP(aws-mcp 같은 것)처럼 몸통을 세트에 못 담던 구멍을 메운다:
몸통을 못 담으면 주소로 가리키면 된다. 세트는 도구 모음이라 **여러 저장소를 걸칠
수 있다**(저장소마다 참조 하나). 받는 사람이 세트를 적용할 때, 그 저장소가
**본인 손으로 등록돼 있어야만** 설치된다 — 등록은 신뢰 표명이라 세트가 대신 하지
않는다. 안 물려 있으면 "이 저장소를 먼저 등록하세요: `pouch repo add …`"라고
안내하고 그 항목만 건너뛴다(나머지는 계속 적용). 같은 이름이 딴 주소로 물려
있으면 조용히 그걸로 설치하지 않고 알린다.

세트는 **가리키는 목록(선곡표)**이다: 이름·설명·어떤 역할에 맞는지 + 항목들
(어디서 가져올지, 그중 무엇을 올릴지)을 적은 JSON 파일 하나. 도구를 새로 담는
통이 아니라서, 가져오기·설치는 전부 기존 카탈로그 경로를 그대로 탄다.
내장 세트 외에 `~/.pouch/sets/`에 같은 형식의 파일을 두면 함께 읽히고, 같은
이름이면 사용자 것이 이긴다.

세트를 옆 사람과 주고받는 첫 길도 이제 있다 — 남이 준 세트 파일은
`pouch set import <경로>`로 들인다. **들이기만으로는 아무것도 실행·설치되지
않는다**(세트는 목록일 뿐) — 실제 적용과 동의는 `set apply`가 맡는다. 남이 준
파일인 만큼 의심도 여기서 한다: 이름에 `../` 같은 탈출 꼴이 박혀 있으면
들이기를 거부한다(세트 폴더 밖 아무 데나 쓰는 것 방지, 2026-07-17 오너 리뷰에서
실증 후 차단).

팀 단위로 주고받는 길(raft)의 받는 쪽도 첫 조각이 나왔다 — 팀이 세트 파일을
모아두는 git 저장소(레지스트리)를 `pouch set pull <git주소>`로 통째로 당겨온다
(두 번째부터는 주소 없이 `pull`만). 당겨온 세트는 내 것과 **딴 서랍**
(`~/.pouch/registry/`)에 살고 `작성자/이름`으로 목록에 잡히며, 이름이 겹치면
**내 것이 이긴다**. 이것도 들이기만 한다 — 적용·동의는 언제나 `set apply`.
왜 매체가 git인지, 무엇이 절대 밖으로 안 나가는지(기억·경계·사용기록)는
[RAFT-DESIGN.md](RAFT-DESIGN.md) — 2026-07-17 오너가 "v0 매체, 엔지니어 팀
대상"으로 좁혀 락했다.

- 세트 형식·읽기: [src/pouch/sets/model.py](../src/pouch/sets/model.py) ·
  적용: [src/pouch/sets/apply.py](../src/pouch/sets/apply.py) ·
  내장 세트: [src/pouch/sets/builtin/](../src/pouch/sets/builtin/) ·
  입구: `pouch set list` / `pouch set apply <이름>`
  ([src/pouch/sets/commands.py](../src/pouch/sets/commands.py))

다시 돌려도 안전하다(같은 답이면 같은 결과, 기존 기억은 덮어씀).

## ④ 기록된다 — 세션 중 자동

여기서부터는 내가 할 일이 없다. Claude Code를 그냥 쓰면:

- 세션이 시작될 때마다 기억 목차가 주입된다. 목차만 주입하고 본문은 필요할 때
  에이전트가 `pouch memory recall <검색어>`로 꺼내온다 (매 세션 전문을 다 넣으면
  대화가 무거워지므로). 단 **경계(boundary) 타입 기억만은 본문까지, 맨 위에** 들어간다 —
  "어디까지 알아서 해도 되는가"는 요약이 아니라 원문으로 알아야 하기 때문이다.
  이 주입엔 **새 기억을 남기는 법**도 실린다 — 에이전트가 작업 중 배운 것을
  `pouch memory add`로 직접 적게 하는 지침이라, 네이티브 메모리를 대체해도 쓰기
  길이 끊기지 않는다. 지침은 종류에 따라 두 갈래다: 덜 민감한 것(진행 중인 일·
  링크)은 **문 앞(`--pending`)으로 편하게**, 민감한 것(나·피드백·경계)은 **사용자가
  말했거나 확인한 자리에서만** — 잘못 알아들은 한마디가 상시 규칙으로 굳는 것을
  막는다. ([src/pouch/memory/context.py](../src/pouch/memory/context.py))
- 도구를 쓸 때마다 `{어떤 도구, 언제}` 한 줄이 `usage.jsonl`에 덧붙는다.
  이 파일은 **덧붙이기만 하고 절대 고쳐 쓰지 않는다** — 이력은 사실의 기록이라
  수정하면 안 되기 때문이다. ([src/pouch/evolution/usage_log.py](../src/pouch/evolution/usage_log.py))
- 다만 이 일기장이 무한히 커지지 않도록, 아주 오래된(180일 밖) 기록은 `pouch evolve`가
  돌 때 **요약으로 접힌다**(compaction). 개별 시각은 흐려지되 "이 도구를 총 몇 번 썼다"는
  누적은 `usage-summary.json`에 영구 보존된다 — 사람이 옛 기억은 흐려져도 습관은 남는 것과
  같다. 접기만은 예외적으로 파일을 다시 쓴다(정상 로깅은 여전히 덧붙이기만).
  ([src/pouch/evolution/compaction.py](../src/pouch/evolution/compaction.py))

한 가지 꼬임과 그 해법: Claude Code는 플러그인에서 온 도구를 실행할 때
`plugin_아무개_exa`처럼 **다른 이름**으로 부른다. 장부에는 `exa`로 적혀 있는데
기록에는 `plugin_아무개_exa`로 쌓이면 둘이 같은 도구인 걸 모르게 된다.
그래서 장부의 각 도구에 **별명(alias)** 칸이 있고, 집계할 때 별명을 정식 이름으로
접어서 센다. ([src/pouch/evolution/aggregate.py](../src/pouch/evolution/aggregate.py))

`pouch`를 인자 없이 치면 지금까지의 그림을 한 화면으로 보여준다 —
뭐가 담겼고, 최근 7일간 뭘 썼고, **손에 맞은 핵심 도구**는 무엇이고, 연결은 살아있는지.
([src/pouch/status.py](../src/pouch/status.py))

기간을 넓혀 보려면 `pouch report [--days N]` — "이번 주/달 뭘 많이 쓰고, 뭐가 안 쓰여
닳고, 주머니 밖에서 뭘 자주 쓰는지"를 시간축으로 보여준다(status의 시간축 확장, 읽기전용).
([src/pouch/report.py](../src/pouch/report.py))

### 이번 목표 한 줄 — 명령은 남았지만 자동 주입은 내렸다 (2026-07-21)

> **표면에서 내림 — 코드는 살아있다.** 아래 명령(`set`/`show`/`clear`)은 그대로
> 쓸 수 있다. 다만 세션 시작에 목표와 되짚는 규약을 **자동으로 실어 보내던 것**은
> 내렸다. 왜: 며칠 실제로 써보니 도움이 안 됐다. 발동이 전적으로 모델의 자율 준수에만
> 걸려 있어(강제하는 훅이 없어) 기능이 따로 놀았고, 그러면서도 매 세션 시작마다
> 규약 40여 줄의 맥락 비용을 냈다. **떨어진다 ≠ 삭제된다** — 명령과 render.py는
> 지우지 않고 남겨, 나중에 훅으로 발동을 강제할 방법이 생기면 다시 올린다.

원래 문제의식은 이랬다. 대화가 길어지면 앞부분이 접히고, "원래 뭘 하려던 거였지"가
흐려진다. 에이전트는 남은 조각으로 목표를 다시 요약하고, 그 요약을 또 요약하면 조금씩
어긋난다. 그래서 작업 시작에 목표를 **한 줄로 박아두는** 명령을 뒀다.

```text
pouch checkpoint set "앵커를 프로젝트 칸으로 옮긴다"
pouch checkpoint show     # 박아둔 한 줄만 그대로 뱉는다
pouch checkpoint clear    # 작업이 끝나면 비운다
```

지금은 이 한 줄이 세션 시작에 저절로 실려 나가지 않는다 — `show`로 직접 불러야 나온다.
예전엔 이 한 줄과 함께 **되짚는 규약**(언제·어떤 모양으로 중간 점검을 내놓을지, 네 칸짜리
`⟦ALIGN⟧` 블록)이 기억 목차와 같은 통로로 주입됐지만, 그 주입을 통째로 내렸다.

**목표는 폴더마다 따로 산다** (이 규칙은 저장 구조라 그대로 살아있다 — `show`로 볼 때 적용된다).

| 어디서 박았나 | 어디에 저장되나 | 어디서 보이나 |
| --- | --- | --- |
| 프로젝트 폴더 안 | `<프로젝트>/.pouch/anchor.json` | 그 프로젝트에서만 |
| 프로젝트 밖 | `~/.pouch/anchor.json` | 프로젝트 밖에서만 |

프로젝트 안인데 그 프로젝트에 박아둔 목표가 없으면 **"목표 없음"으로 시작한다** —
글로벌 것을 대신 읽지 않는다. 예전엔 목표 자리가 온 컴퓨터에 하나뿐이라, A 프로젝트에서
박은 목표가 B 프로젝트 작업 중에 그대로 튀어나왔다(2026-07-21에 실제로 일어났다).
한 번이라도 "없으면 글로벌 것이라도"를 허용하면 그 사고가 되살아나므로, 대신 읽는
길 자체를 안 만들었다.

([src/pouch/checkpoint/anchor.py](../src/pouch/checkpoint/anchor.py) — 목표 저장,
[render.py](../src/pouch/checkpoint/render.py) — 되짚는 규약 본문. 지금은 주입에서 빠져

## ⑤ 제안받는다 — `pouch evolve`

쌓인 기록을 근거로 pouch가 제안한다. **제안만 한다. 동의 없이는 아무것도 안 움직인다.**
이것이 pouch 전체에서 가장 강한 원칙이다.

### 제안 전에 evolve가 조용히 하는 두 가지 (자동, 동의 불필요)

`pouch evolve`는 제안 목록을 띄우기 전에 두 가지 정리를 먼저 한다. 둘 다 사실을
정돈하는 일이라 동의를 안 받는다("자동으로 움직이는 건 기록뿐"의 범위):

1. **오래된 기록 접기(compaction)** — 180일 밖 사용 기록을 요약으로 접어 로그가
   무한히 커지는 걸 막는다(습관 누적은 보존).
2. **소스 → 장부 진입(reconcile)** — ①에서 예고한 그 동작이 **여기서 실제로
   일어난다.** 소스에 재워둔 것 중 이번에 실제로 쓴 게 있으면, evolve가 찾아서
   장부로 올린다. 화면에 `📥 실제로 쓴 도구 N개가 카탈로그에 진입했어요`로 뜬다.

> **그래서 "import했는데 왜 조용하지?"의 답이 여기 있다.** import는 소스에
> 적어두기만 하고, 그 도구를 한 번 쓴 뒤 `pouch evolve`가 돌 때 비로소 장부로
> 진입한다. 진입 = pouch가 그 도구를 "네 도구"로 챙기기 시작하는 순간(①의 표 참조).

이 진입은 **단조**다 — 한번 들어오면 소스를 다시 안 건드린다. 아래 drop이 도구를
"내려도" 장부·소스는 그대로라, 진입과 drop이 서로 밀당하며 진동하지 않는다.

### 그다음, 한 화면에 네 종류의 제안이 나온다

| 제안 | 신호 | 뜻 |
| --- | --- | --- |
| 🌊 내리자 (drop) | 설치 후 한 번도 안 씀, 또는 오래 안 씀(기본 30일) | 표면에서 빼자 |
| 🔗 올리자 (attach) | 장부에 있는데 표면에 없고, 최근 7일 내 쓴 흔적 | 다시 표면에 올리자 |
| 🆕 확인할 기억 | 가볍게 담겨 대기 중인 기억(pending) | 진짜 기억으로 승격할까? |
| 🧹 정리할 기억 | 낡음이 감지된 기억 | 목차에서 내릴까? (아래 참조) |

숫자 두 개가 왜 7과 30인지: "안 쓰면 내리자"의 기준(30일)보다 "다시 쓰네, 올리자"의
기준(7일)이 훨씬 짧아서, **같은 도구가 내려갔다 올라갔다를 반복하는 일이 구조적으로
불가능**하다. 우연이 아니라 의도된 간격이다.

한 가지 알아둘 예외: **사용 신호가 안 찍히는 종류(훅·규칙·에이전트)는 "내리자"
제안 대상에서 빠진다.** 훅은 기록에 안 남을 뿐 매 사건마다 일하고 있어서,
"기록 없음 = 안 쓰임"이 성립하지 않기 때문이다. 이런 종류를 내리고 싶으면
`pouch catalog uninstall <id>`로 손수 내린다(카탈로그·개인화는 남는다).

또 하나: **핵심 도구(지속·빈도로 손에 맞은 것)도 "내리자"에서 빠진다.** 오래 걸쳐
꾸준히 써온 도구는 잠깐의 공백에도 흔들리지 않게 보호한다(기억에서 weight 높은 것이
위생 제안에서 빠지는 것과 같은 정신). 무엇이 핵심인지는 `pouch`·`pouch report`가 🪨로 보여준다.

### 💡 이거 써봐 — 잘 쓰는 것과 비슷한 걸 대기실에서 찾아준다

제안 화면 끝에 하나 더 붙는다. 음악 앱의 "이 노래 좋아하면 이것도" 같은 것 —
**반복해서 쓰는 도구**(2회 이상, 표면에 있든 없든)를 기준으로, 설명·태그의 단어가
겹치는 도구를 후보로 보여준다. 왜 비슷한지(겹친 단어)를 실재 근거로 함께 내고,
써볼지는 사용자가 고른다(`pouch catalog install <이름>`). 안내만 — 자동 설치 없음.

처음 만들 때(2026-07-12)는 두 겹이 좁아서 **한 번도 안 떴다**(2026-07-22 발견·수리):

- **기준이 구조적 공집합** — "썼는데 표면에 없는 것"을 기준으로 삼았는데, 쓰려면
  표면에 있어야 하니 잘 쓰는 도구는 전부 탈락했다.
- **후보 풀이 장부만** — sweep이 대기실(소스)에 472개를 재워놨는데 비교는 장부
  9개 안에서만 했다.

배승도 락(2026-07-22)으로 둘 다 폈다: 기준 = 반복 사용 통계(별칭 접기·요약 포함),
풀 = 장부 + 대기실. 대기실도 sweep이 실측한 "이미 깔린 것"이라 지어내기 금지 원칙
안이다 — "남들이 많이 쓴다" 같은 바깥 마켓 신호는 여전히 raft(Phase 5) 뒤.
화면 소음은 출구에서 막는다: 보여줄 게 있는 기준 상위 3개까지, 설명은 한 줄로 자름.

풀에 겹이 하나 더 생겼다(2026-07-24, Phase 4.8 ③): **등록한 저장소의 색인.**
장부 → 대기실 → 저장소 순으로 넓어지고, 저장소 후보는 정체가 `<저장소>/<도구>`라
화면에 "○○ 저장소가 담고 있음"이 함께 나온다 — pouch가 고른 게 아니라 그 저장소를
만든 **사람의 큐레이션을 전달**하는 것임이 남는다. 이미 장부·대기실에 있는 도구의
저장소 사본은 다시 안 권한다(같은 걸 두 번 권하는 소음 방지).

솔직한 한계: 비슷함은 영문 단어 겹침이라 **한글 설명 도구(seonbi·nunnopi 등)는
기준이 돼도 후보가 안 붙는다**. 대기실이 거의 영어 설명이라 한글 토큰을 살려도
만날 상대가 없어, 한글 도구가 늘 때까지 보류.

### 떨어진다 ≠ 삭제된다

evolve가 도구를 "내려도" 일어나는 일은 표면에서 빠지는 것뿐이다.
장부의 항목, 포스트잇(overlay), 경계 설정은 전부 그대로 남는다.
다시 올리는 공식 입구는 `pouch catalog install <id>` — 내렸던 도구가 개인화까지
고스란히 돌아온다. 지우는 것은 별도의 명시적 행동이고, 조용히 일어나지 않는다.

예외 하나: 플러그인 시스템이 직접 관리하는 도구는 pouch가 올리고 내리지 않는다
(이중 관리가 되면 서로 꼬이므로). 그런 항목은 "플러그인이 관리 중"으로 **관측만** 한다.

- 후보 계산(순수 계산 — 파일도 시계도 안 만짐, 테스트하기 쉬운 형태):
  [src/pouch/evolution/candidates.py](../src/pouch/evolution/candidates.py),
  [src/pouch/evolution/attach.py](../src/pouch/evolution/attach.py)
- 제안 → 동의 → 실행의 흐름: [src/pouch/evolution/orchestrate.py](../src/pouch/evolution/orchestrate.py)

## 기억의 생애주기 — 도구와 같은 모양

기억도 도구와 똑같은 철학으로 돈다: 들어올 땐 가볍게, 나갈 땐 신중하게, 지우는 건 시끄럽게.

### 기억 한 건의 모양

기억 하나 = 마크다운 파일 하나. 머리(이름·타입·범위·요약)와 본문으로 되어 있다.
사람이 열어 읽을 수 있고, git으로 관리할 수 있고, 그대로 백업할 수 있다 —
일부러 데이터베이스를 안 썼다. ([src/pouch/memory/model.py](../src/pouch/memory/model.py))

타입은 다섯: `user`(내가 누군지) / `project`(진행 중인 일) / `reference`(링크·경로) /
`feedback`(일하는 방식에 대한 지적) / `boundary`(자율성의 경계 — 유일하게 본문까지 주입).

### 남의 기억을 넘겨받기 — `pouch memory adopt`

Claude Code에도 자체 메모리가 있다(`~/.claude/projects/<프로젝트>/memory/`, 세션마다
자동 로드). `pouch memory adopt`는 그걸 pouch로 **옮겨온다** — 타입이 자리·계층을
정한다: `user`·`feedback`는 전역 주입, `reference`는 프로젝트 주입, `project`(대개 지난
세션로그)는 **주입 안 함**(리뷰 대기, 날짜 박힌 건 보관). 네이티브가 생명주기 없이 모든
걸 매 세션 주입하던 것을, pouch의 "안정 핵심만 주입"으로 복원하며 청소한다.

이관은 **복사**라 원본을 안 지운다. 기본은 옮기기만 하고 **네이티브는 안전망으로 남긴다**
— 완전 대체(네이티브 자동로드 끄기)는 `--disable-native`로 명시 선택한다(그때는 세션
주입에 심어둔 "기억하려면 `pouch memory add`" 지침이 에이전트의 쓰기 길을 잇는다).
([src/pouch/memory/adopt.py](../src/pouch/memory/adopt.py))

### 세 가지 상태 — 책상, 문 앞, 서랍

**세 상태 모두 파일로 저장은 된다.** 갈리는 것은 "기록하느냐"가 아니라 **"매 세션
에이전트에게 자동으로 주입되느냐"** 하나뿐이다:

| 상태 | 비유 | 저장됨? | 매 세션 주입? |
| --- | --- | --- | --- |
| `indexed` | 책상 위 | ✅ | ✅ 목차(`MEMORY.md`)에 실려 매 세션 들어감 |
| `pending` | 문 앞에서 대기 | ✅ | ❌ 확인 전까지는 주입 안 됨 |
| `archived` | 서랍 속 | ✅ | ❌ 주입 안 되지만 recall로 부르면 나옴 |

이 3단계가 푸는 긴장이 있다. 기억을 쉽게 쌓게 만들수록(저마찰) 쓰레기도 쉽게
쌓인다. 해법: 가볍게 들어오는 것(`pouch memory add --pending`)은 저장은 하되 일단
**문 앞(pending)**에 세워두고, 책상에 올리는 것(promote)만 내 확인을 거친다.
그러면 쓰레기가 쌓여도 문 앞에 쌓이지, 에이전트가 매번 읽는 책상에는 안 쌓인다.
저마찰의 뜻은 "막 담아도 안전하다 — 확인 전엔 에이전트를 오염시키지 못하니까"다.

단 타입마다 문의 무게가 다르다. `project`·`reference`는 확인 없이 문 앞에 설 수
있지만, **`feedback`·`boundary`·`user`는 pending 우회를 코드가 막는다**(반드시
확인을 거쳐야만 들어옴) — 한 번 잘못 알아들은 지적이 "매 세션 주입되는 상시 규칙"이
되는 사고를 막기 위해서다. 문서 약속이 아니라 코드가 막는다는 점이 중요하다
("규칙은 코드로" 원칙의 실물 예시).

### 기억은 나이가 아니라 진실로 낡는다

"오래됐으니 내리자"는 `project` 타입에만 통한다(일은 끝나니까). 타입별로 낡음의
신호가 다르다:

- `project` → 만료 (적힌 일이 끝났는가)
- `reference` → 생존성 (가리키는 경로·URL이 아직 살아있는가)
- `boundary` → **위생 대상에서 제외** (안 걸리는 금지 규칙은 낡은 게 아니라 제 일을 하는 중)
- `feedback`·`user` → 모순이 유일한 신호인데 감지는 아직 없음. **알려진 구멍:
  현재 이 둘은 명시적으로 지우기 전엔 안 내려간다.**

낡음이 감지돼도 자동으로 안 내린다 — evolve가 제안하고 내가 동의해야 서랍으로
간다. `weight`(중요도)가 높은 기억은 제안 대상에서도 빠진다(면역).

- 낡음 신호 계산: [src/pouch/memory/hygiene.py](../src/pouch/memory/hygiene.py) /
  경로·URL 생존 확인: [src/pouch/memory/liveness.py](../src/pouch/memory/liveness.py)
- 문 앞 대기열: [src/pouch/memory/pending.py](../src/pouch/memory/pending.py)
- 상태 전이(승격·강등)와 목차 갱신을 한 곳에서:
  [src/pouch/memory/store.py](../src/pouch/memory/store.py)

### 경계(boundary)에 관한 안전 비대칭 하나

경계는 **자기 명령**을 가진다 — `pouch boundary add/list/remove`로 걸고(방향 필수),
한눈에 보고(스코프·방향별), 지운다. `pouch init`도 흔한 안전 경계(force-push·prod·
자격증명)를 제안한다(고른 것만 담김). 저장은 여전히 기억이라, 아래 규칙이 그대로 적용된다.

경계 기억엔 기울어진 규칙이 있다: **금지(deny)는 넓게, 허용(allow)은 좁게.**
그리고 허용의 누수는 코드가 아니라 **저장 위치가** 막는다 — "이 작업은 알아서 해도 됨"
같은 허용은 프로젝트 범위로 저장돼 그 저장소의 `.pouch/`에만 존재하므로,
다른 프로젝트에서는 물리적으로 읽힐 수가 없다.

또 하나의 의도된 결정: pouch는 경계를 **강제하지 않는다** (에이전트의 도구 호출을
막는 차단기를 안 단다). 진짜 강제는 인프라(IAM 같은 권한 체계)의 일이고, pouch가
차단을 흉내 내면 "pouch가 막아주니까 괜찮겠지"라는 거짓 안전감이 생기는데,
그게 차단이 없는 것보다 위험하기 때문이다. pouch의 경계는
**에이전트가 알고 존중하게 만드는 층**이다.

## 잃었을 때 — 백업과 복원

`~/.pouch`는 쓸수록 쌓이는 만큼 잃으면 아픈 단일 장애점이다. 두 명령이 지킨다:

- **`pouch backup`** — 전역 주머니(`~/.pouch`)를 타임스탬프 붙은 아카이브 하나
  (tar.gz)로 싸서 `~/pouch-backups/`에 둔다. 비파괴적이라 바로 실행된다.
- **`pouch restore <아카이브>`** — 주머니를 그 백업 시점으로 되돌린다. 현재 상태를
  백업 시점으로 **교체**(replace)하되, 덮기 전에 지금 상태를 자동 스냅샷으로 남긴다
  (`pre-restore-…`). 되돌리기의 되돌리기다 — settings.json 고칠 때 `.bak` 남기는 것과
  같은 정신. 백업 후 생긴 파일은 사라지지만 스냅샷 안에 살아있어 소실이 아니다.

두 가지 의도된 결정:

- **글로벌만 담는다.** 프로젝트별 `.pouch/`엔 민감정보(경계·클라이언트 정보)가 섞일 수
  있어 클라우드에 안 보낸다. 그건 각 저장소의 git이 맡는다. → 데이터 유출을 구조로 회피.
- **코어는 목적지를 모른다.** "무엇을 싸서 어떻게 되푸나"(코어)와 "어디로 보내나"(어댑터)를
  갈랐다. v0 목적지는 로컬 폴더뿐이고, S3(기술자)·구글드라이브(비기술자)는 인증·API만 다른
  얇은 어댑터로 같은 코어 위에 얹는다. 이 백업 코어는 Phase 5 팀 공유(raft)의 뿌리이기도
  하다 — 백업은 나에게 되돌리기, 공유는 남에게 주기, 데이터 형태가 같다.

- 싸기/되풀기: [src/pouch/backup/core.py](../src/pouch/backup/core.py) ·
  로컬 어댑터 + 복원 안전장치: [src/pouch/backup/local.py](../src/pouch/backup/local.py) ·
  CLI: [src/pouch/backup/commands.py](../src/pouch/backup/commands.py)

## 전 과정을 관통하는 규칙 셋

1. **제안만 한다.** 도구를 내리는 것도, 기억을 서랍에 넣는 것도, 전부 제안 → 내 동의 → 실행.
   자동으로 움직이는 것은 기록(로그)뿐이다.
2. **떨어진다 ≠ 삭제된다.** evolve의 "내림(drop)"은 표면에서 빼는 것이지 장부에서
   지우는 게 아니다. 되돌리는 공식 입구가 항상 있다. (예외는 `catalog migrate`
   하나 — 이건 장부에서 소스로 되돌리는 *이동*이라 장부 파일을 지운다. 그래도
   삭제가 아니라 강등이고, 조용히 안 일어난다: 사용자가 직접 부르고, 적용 전
   백업을 뜨고, 재사용하면 다시 진입한다. "시끄러운 행동"의 조건을 다 지킨다.)
3. **설정은 대신 바꾸되 투명하게.** pouch가 `settings.json`을 대신 고쳐주지만
   항상 설명·동의·백업·복구 수단이 따라온다. `--yes`로 건너뛸 수는 있어도
   백업은 건너뛰지 않는다.

# 🦦 기억으로 뭘로 차별화하나 — 남의 메모리 시스템과 pouch의 자리

> **이 문서는 같이 개발하는 사람과 "pouch의 기억이 기존 메모리 시스템과 뭐가
> 다른가"를 정하려고 쓴다.** 저자의 개인 스택(Claude 기본 메모리 · hindsight ·
> Obsidian)을 뜯어 비교하되, **결론은 저자 한 명이 아니라 "남도 쓴다"는 전제**로
> 내린다 — 그래서 개인 스택은 *기준선*이 아니라 *선행 사례(prior art)*로만 읽는다.
>
> 읽는 법은 [HOW-IT-WORKS.md](HOW-IT-WORKS.md)와 같다 — 막히는 단어는 표시해뒀다가
> [GLOSSARY.md](GLOSSARY.md)로 넘긴다.

---

## 한 줄 결론

pouch의 기억은 **"저장소"로 경쟁하지 않는다.** 저장은 이미 아무 시스템이나 한다.
pouch의 자리는 **능동 주입 + 자율성 경계 + 자가정리 생명주기를, 여러 에이전트에
걸쳐** 제공하는 *레이어*다. 이 셋이 겹치는 시스템은 (조사한 범위 안에) 없다.

---

## 비교 대상 — 그리고 "남도 쓴다"가 바꾸는 것

저자 개인 스택은 셋이다. 하지만 **일반 사용자가 다 가진 게 아니다** — 이 구분이
차별화 방향을 통째로 바꾼다.

| 시스템 | 일반 사용자가 가지고 있나 | 이 문서에서의 지위 |
|---|---|---|
| **Claude Code 기본 메모리** | Claude Code 쓰면 **있음**(단 호스트마다 다름) | 진짜 경쟁/중복 상대 |
| **hindsight MCP** | 거의 **없음**(직접 서버 띄운 사람만) | 선행 사례 · *선택적* 통합 후보 |
| **Obsidian wiki** | 거의 **없음**(second brain 쓰는 사람만) | 선행 사례 · *선택적* 통합 후보 |

> **핵심 전환:** hindsight·Obsidian은 pouch가 **의존하면 안 되는** 시스템이다.
> 남의 노트북엔 없으니까. 이들은 "있으면 꽂아 쓰는 백엔드"(회상 엔진·내보내기
> 목적지)일 뿐, pouch 코어는 이들 없이 혼자 서야 한다. 앞선 초안에서 "회상은
> hindsight에 위임" 같은 말은 *저자 한정* 최적화였고, 일반화하면 틀린다.

## pouch에만 있는 것 (방어 가능한 차별점)

1. **능동 주입 × 멀티 호스트.** 나머지는 전부 *pull*이거나 *Claude 전용 push*다.
   pouch만 "세션 시작에 여러 에이전트로 밀어넣는" 레이어다. 일반 사용자가 Codex나
   Kiro를 쓰면 **거기엔 Claude 같은 네이티브 메모리가 아예 없다** — pouch가 그
   빈자리를 채우는 유일한 공통분모가 된다. 이게 가장 큰 구조적 해자.

2. **Boundary(자율성 경계) 타입.** 방향성(ALLOW/ASK/DENY) + 최상단 강조 주입 +
   `[DENY]` 라벨이 산문보다 우선 + 도구와 함께 사는 생명주기. hindsight의
   `directive`가 개념상 가깝지만 그건 pull이고, pouch는 push + 안전 우선 정책이다.
   Claude 기본·Obsidian엔 아예 없다. → **킬러 피처 후보 1순위.**

3. **명시적 생명주기 + "기억은 나이가 아니라 진실로 닳는다".** project=만료,
   reference=생존성(죽은 URL), boundary=면역 — 타입별 붕괴 신호를 코드로 박음.
   Claude 기본은 삭제뿐, hindsight는 블랙박스.

4. **도구 카탈로그와의 결합.** boundary가 catalog 도구와 묶여 drop gate가 방향으로
   갈린다. 순수 메모리 저장소가 아니라 하네스의 일부 — 이건 메모리 단품이
   흉내 못 낸다.

## pouch가 약한 것 (남이 이미 잘함)

| 약점 | 누가 잘하나 | 일반 사용자 관점의 처방 |
|---|---|---|
| 시맨틱 회상 | hindsight | 코어는 키워드로 자립. 회상을 **플러그형 백엔드**로 열어, 있으면 hindsight를 꽂게(의존 아님) |
| 사람 큐레이션·영속 지식 | Obsidian | pouch는 여기 경쟁 안 함. 내보내기 목적지로 **선택적** 연동만 |
| 세션→기억 자동 축적 | (셋 다 못함 — 열린 문제) | 로드맵 미완. 여기를 풀면 셋 전부를 앞선다 |

## ⚠️ 진짜 중복 상대는 Claude 기본 메모리다

pouch 메모리는 **저장 형태·타입·MEMORY.md 인덱스가 Claude 기본과 거의 판박이**다
(코드 주석도 "Claude 네이티브 메모리 분류와 호환된다"고 못박음). 그래서 Claude
Code 사용자가 둘 다 켜면 **겹치는 MEMORY.md가 두 벌** 생긴다.

여기서 "남도 쓴다"가 결정을 뒤집는다. 호스트마다 네이티브 메모리 유무가 다르다:

| 호스트 | 네이티브 메모리 | pouch의 역할 |
|---|---|---|
| Claude Code | 있음 | **중복 위험** — 관계 정리 필요 |
| Codex | 없음(빈약) | pouch가 **유일한** 기억 |
| Kiro | 파일 steering만 | pouch가 채움 |

→ pouch를 "각 호스트의 메모리와 경쟁하는 또 하나의 저장소"로 두면 Claude에선
중복, 나머지에선 무관해진다. 대신 **"호스트를 가로지르는 하나의 기억 + 정책
레이어"**로 두면, Claude에선 네이티브를 흡수/상위화하고 나머지에선 없던 걸 채운다.

## 포지셔닝 옵션

| 옵션 | Claude 기본 메모리를... | 남도 쓴다 관점 평가 |
|---|---|---|
| **A. 대체** | 끄고 pouch가 그 자리를 먹음 | 멀티호스트·경계·생명주기로 상위호환. 단 Claude 자동로드를 훅으로 재현해야 |
| **B. 상위 레이어** | 파일을 **읽어와** 주입·경계·생명주기만 pouch가 얹음 | 저장 중복 제거 + "여러 에이전트 + 정책"이라는 순수 차별만 남음 |
| **C. 병존(현재)** | 각자 따로 | 중복 유지비. 일반 사용자에게 혼란. 비추 |

### 추천 방향 (초안, 미확정)

- **차별화는 저장으로 하지 않는다.** pouch의 세 축(주입·경계·수명)을 더 깊게 판다.
- **Boundary를 킬러 피처로 밀어붙인다.** 어느 메모리 시스템도 "허용/확인/금지 +
  방향성 + 안전 우선 주입"을 1급으로 안 다룬다. 가장 방어 가능한 해자.
- **코어는 자립, 외부는 선택적 통합.** hindsight(회상 백엔드)·Obsidian(내보내기)은
  *있으면 꽂는* 어댑터로. 없어도 pouch는 완전 동작해야 한다 (남의 노트북엔 없다).
- **Claude 기본과의 관계는 A(대체)를 제안** (오너 확인 전) — 네이티브를 `autoMemoryEnabled: false`로
  완전히 끌 수 있음이 확인돼(읽기·쓰기 둘 다 중단) 깨끗한 대체가 가능하다. 게다가
  네이티브는 생명주기가 없어 세션로그가 무한 적재되는 실측 문제가 있어(한 프로젝트에
  53파일 사례), 위에 얹기(B)보다 아예 대체하며 청소하는 게 낫다. 설계는
  → [MEMORY-REPLACE-DESIGN.md](MEMORY-REPLACE-DESIGN.md).

## 다음에 정할 열린 결정들

> ⚠️ **아직 잠기지 않았다 — 오너 확인 전.** 아래 "닫힘"은 같이 개발하는 사람이
> 제안하며 닫은 것으로 적었지만, 오너(배승도)가 다시 말로 풀어 확인한 적이 없다.
> pouch 원칙 "잠그기 전에 오너가 다시 말해 확인한다"에 따라 **확인 전까지는 모두
> 열린 제안으로 읽는다.** 코드가 이미 그 방향으로 갔더라도 그게 결정이 굳었다는
> 뜻은 아니다.

1. **A vs B** — 제안: A(대체). **열림 — 오너 확인 대기.** 근거·설계는
   [MEMORY-REPLACE-DESIGN.md](MEMORY-REPLACE-DESIGN.md). A가 네이티브의 write 경로까지
   없애서 아래 #4(자동 축적)를 "선택"에서 "필수"로 앞당긴다는 게 제안의 요지.
2. **경계(boundary)를 어디까지 킬러 피처로** — 제안: 1급화 A(최상위 `pouch boundary`
   add/list/remove) + B(init 안전 경계 제안). **열림 — 오너 확인 대기**
   (→ [BOUNDARY-PROMOTION.md](BOUNDARY-PROMOTION.md)). 모델·비강제 철학은 그대로.
   C(별도 저장)·D(강제)는 과설계·철학위반이라 빼자는 제안.
3. **회상 백엔드 추상화** — 키워드/시맨틱/외부(hindsight)를 갈아끼우는 인터페이스를
   지금 그을지, 나중으로 미룰지.
4. **세션→기억 자동 축적** — 셋 다 못 푼 열린 문제. pouch가 여기서 앞설지, 아니면
   경계/주입에 집중할지.

> 이 문서는 결정을 *기록*하는 게 아니라 *열어두는* 문서다. 위 넷을 하나씩 닫을
> 때마다 여기(또는 ROADMAP)에 근거와 함께 남긴다.

# 🦦 raft 설계 — 개인에서 팀으로 (Phase 5)

> 한 사람의 좋은 돌이 무리로 번진다. 해달들이 잘 때 떠내려가지 않으려 서로 손을
> 잡고 무리를 이루는 걸 *a raft of otters*라 부른다 — 각자 띄운 주머니가 옆
> 사람과 묶여 한 뗏목이 되는 것. 이 문서는 그 첫 정책 락이다.

## 확정된 방향 (2026-07-16)

| 결정 | 선택 | 이유 |
| --- | --- | --- |
| 공유 매체 | **Git 기반 공유 레지스트리** | 새 백엔드 0. 회사가 이미 git 사용. 버전관리·리뷰·인정(star/PR)이 공짜로 딸려온다. export/import가 이미 세트를 "파일"로 만들어(#8·#9) repo = 세트 파일 모음이라 자연스럽다. **로컬 우선** 이념과 일치(clone→로컬) |
| 공유 단위 | **세트(도구 선곡표)만** | 나가는 건 "무엇을 어디서 가져와 무엇을 올릴지". 기억·경계·사용기록은 로컬 전용 |
| 방향 | **명시 행위** | 공유는 pull/publish라는 사람의 결정. 자동 push·자동 편입 없음(개인 우선·동의 원칙) |

### 왜 git인가 (탈락지 근거)

- **호스티드 서버** — 검색·스타·인기 데이터로 UX는 가장 풍부하나, 서버를 짓고
  운영·호스팅하는 부담이 v0엔 과하다. "로컬 우선/새 백엔드 없음"과 충돌.
- **오브젝트 스토리지(S3/GDrive)** — 백업 어댑터를 재사용할 수 있으나, 목록·인정·
  리뷰가 약하고 접근제어를 따로 설계해야 한다. 백업(나에게 되돌리기)엔 맞지만
  공유(남에게 주기, 발견·인정이 핵심)엔 덜 맞는다.
- **파일 전달만** — 인프라 0이지만 *묶임(raft)* 이 안 생긴다: 발견도, 인정 순환도
  없다. 이건 export/import(#8·#9)로 이미 있는 상태 — Phase 5가 아니다.

git은 이 셋의 약점을 동시에 피한다: 서버 없이, 발견·리뷰·인정을 repo가 이미
제공하고, 세트가 파일이라 얹기만 하면 된다.

## 개인/공용 경계 (안전 — 넘지 않는 선)

raft가 켜져도 **개인은 절대 레지스트리로 나가지 않는다.** 이건 이미 구조적으로
참이라(세트는 도구 표면만 담는다), raft는 이 경계를 *새로 만드는 게 아니라 넘지
않음을 계약으로 고정*한다.

| 나가는 것 (공용) | 절대 안 나가는 것 (개인) |
| --- | --- |
| 세트 = source→install 선곡표 (도구가 사는 위치·id) | 기억 전부 (user·feedback·project·reference·boundary) |
| 홈 상대(`~`) 출처 경로 (#8: 절대경로가 안 샌다) | `usage.jsonl` (뭘 얼마나 썼나) |
| match 토큰 (누구를 위한 세트인지) | 프로젝트별 `.pouch/` (민감정보, 각 repo 위임) |

- **경계는 세트 형식이 이미 지킨다.** 세트에 담기는 필드에 기억·사용기록이 없다.
  raft는 "세트만 주고받는다"를 매체로도 강제한다(레지스트리엔 세트 JSON만 올라간다).
- **홈 상대 출처**(#8의 `~/` 접기)가 개인 절대경로 유출을 이미 막는다.

## v0 슬라이스 (구현 순서)

작게 자른다. 발견·인정의 풍부함은 레지스트리가 실제로 쌓인 뒤에 얹는다(지어내기 금지).

### ① `pouch set pull <git-url>` — 받는 쪽 (레지스트리 소비)

레지스트리 repo를 clone/fetch → 그 안의 세트 JSON들을 기존 import 경로(#9)로
`~/.pouch/sets/`에 들인다. import가 이미 검증·정규화·"이 컴퓨터에 없는 출처 보고"를
하므로 재사용. 새 부분은 git clone/pull + repo 안 세트 순회뿐.

- **팀 레지스트리 URL 설정** — 한 번 지정하면(`~/.pouch/`에 저장) 인자 없이
  `pouch set pull`로 재fetch. 여러 세트를 한 번에 들인다.
- 매체가 git이라 **버전·이력·되돌리기가 공짜**(레지스트리 repo가 git이므로).

### ② publish — 주는 쪽 (레지스트리 기여)

v0는 **git-native**: 사용자가 export(#8)한 세트 파일을 레지스트리 repo에
커밋/푸시(또는 PR)한다. pouch는 git plumbing에 손대지 않는다(얇게). 리뷰·인정은
repo의 PR·star가 그대로 한다.

- 편의 헬퍼(`pouch set publish` = 로컬 레지스트리 클론에 세트 복사 + 안내)는
  실사용에서 마찰이 확인되면 다음 슬라이스로. v0는 "export → repo에 올리기".

### ③ (defer) 인정 순환 — 스타 → 기본 온보딩 편입

완성된 세트 공유 → 스타(인정) → 인정받은 세트가 `init`의 기본 온보딩에 편입.
"남들이 많이 쓴다" 인기 데이터는 **레지스트리가 실제로 쌓인 뒤**에만 진짜다
(지금 만들면 지어내기 = 2026-07-07 내장세트 철회의 재발). 매체가 git이라 스타는
repo star를 읽어오면 되지만, 편입 정책은 별도 락이 필요 — v0 이후.

## 안 하는 것 (명시적 defer)

- **호스티드 서버·검색 UI·실시간 인기 데이터** — 레지스트리 실사용 후.
- **자동 push/자동 편입** — 공유는 명시 행위(개인 우선·동의).
- **접근제어·팀 스코프** — v0는 "repo에 접근되면 pull된다"(git 권한에 위임).

## 락된 구조 (2026-07-16 설계 리뷰)

열린 질문 3개를 확정. 셋이 하나로 정합한다(작성자 스코프 → 별도 티어 → 로컬 우선).

### 레지스트리 repo 구조 — 작성자별 폴더

`sets/<author>/<name>.json`. 이름 충돌이 없고, "누구 세트인지"(출처)가 구조에
내장되며, 미래 인정(스타)의 자연 단위가 된다 — 회사 raft는 여러 명이 올리므로
충돌·출처가 실제 문제다. **함의:** 당겨온 세트의 정체(identity)에 작성자가 들어가야
`available_sets`의 이름-키 충돌을 피한다(예: `alice/aws-devops`).

### pull이 git을 부르는 법 — git shell-out

`git clone`/`git pull`을 subprocess로(멈춤 방지 타임아웃). `status.py`의 `_git_revision`
(`git log`)이 이미 쓰는 패턴이라 일관되고, 사용자의 기존 git 인증(SSH/HTTPS)을 그대로
물려받으며, 파이썬 git 라이브러리 의존성을 안 늘린다.

### 로컬 vs 당겨온 세트 — 별도 티어 + 로컬 우선

- **저장 분리** — 당겨온 세트는 `~/.pouch/registry/<author>/<name>.json` 별도 티어에
  캐시. 내가 만든 `~/.pouch/sets/`와 안 섞인다.
- **우선순위** — 이름 충돌 시 **로컬(내 것)이 이긴다**(내장<사용자에서 사용자가 이기는
  개인 우선과 같은 정신). 읽기 순서: registry(남) < builtin < local(나).
- **리프레시 안전** — 재-pull은 registry 티어만 통째 갱신(멱등). 내가 만든 세트는
  절대 안 건드린다.

작성자 네임스페이스 + 별도 티어가 맞물려 충돌을 **구조적으로** 없앤다("규칙은 코드로").

## 구현 시 결정할 사소한 것 (설계 아님)

- `pouch set list`에서 로컬/당겨온 것 표시(그룹핑·작성자 프리픽스).
- 레지스트리 URL 설정 저장 형식(`~/.pouch/`에 어떻게 적을지).
- publish 편의 헬퍼 — v0는 git-native, 마찰 확인되면 다음.

## 뿌리 연결

- **백업 코어와 같은 뿌리** — 백업 = 나에게 되돌리기, 공유 = 남에게 주기.
  "코어는 목적지를 모른다 + 얇은 어댑터" 패턴을 raft도 잇는다(git = 또 하나의 어댑터).
- **import 관문과 같은 정신** — 전량 자동 진입 안 함. pull도 "들이기만" 하고
  실제 적용(표면 올리기)은 `set apply`가 동의받아 한다.

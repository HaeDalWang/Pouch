# 🗂️ pouch 백로그

> **ROADMAP은 확정된 순서, 백로그는 후보 풀이다.** 여기 항목이 채택되면 ROADMAP의
> Phase로 승격되고, 여기선 지워진다. 이 문서도 닳고 붙고 떨어진다.

## 채택 렌즈 — "pouch의 약속이 거짓말이 되는가"

모든 항목엔 **걸린 약속** 칸이 있다. 약속이 *깨지는* 것 = 필수, 약속이 *커지는* 것 = 있으면 좋겠다.

| 약속 | 뜻 |
| --- | --- |
| 🪨 맞춰짐 | 나에게 맞춰진다 |
| 🌊 진화 | 쓸수록 진화한다 (닳고·붙고·떨어진다, 떨어져도 개인화는 남는다) |
| 🦦 묶임 | 옆 사람과 묶일 수 있다 |
| 🔧 규칙은 코드로 | **pouch 자신의 내부 불변식**을 코드 제약으로 박는다 (예: vendored body 불변). 사용자가 에이전트에게 거는 규칙(boundary)은 이 원칙의 대상이 **아니다** — [D1] 참조 |

## 결정 (백로그가 아니라 이미 정한 것 — 흔들리지 않기 위해 기록)

### D1. boundary는 의도적으로 강제하지 않는다 (2026-07-05)

boundary는 주입(에이전트가 알고 존중하게) + 제안까지만. PreToolUse 하드 차단은 넣지 않는다.

- **층 구분**: "규칙은 코드로"는 pouch 내부 불변식용 원칙이다. boundary는 사용자가
  에이전트에게 거는 규칙 — 다른 층이고, 섞으면 안 된다.
- **거짓 안전감**: 진짜 강제가 필요한 prod 보호는 더 낮은 층의 일이다 — IAM, SCP, OPA.
  pouch가 앱 레이어에서 차단을 흉내 내면 "pouch가 막아주니까 IAM 느슨해도 되겠지"라는
  거짓 안전감을 만든다. 그게 차단 없음보다 위험하다.
- 누가 "boundary가 왜 안 막아?"라고 물으면: **막는 건 인프라 층 책임, pouch의 boundary는
  에이전트가 알고 존중하게 만드는 층**이라고 답한다.

### D2. 당분간 Claude Code 우선, 중립화는 마킹만 (2026-07-05)

에이전트 중립 추상화 레이어는 넣지 않는다(YAGNI). 대신 Claude-specific 결합 지점
(hook 연결, `~/.claude/plugins/` 경로 가정, settings.json 스키마 등)에 마커 주석을
남긴다 — 구조 슬롯이 아니라 **문서화된 부채 마커**라 지금 박아도 안 비싸고, 수요가
생겼을 때 "어디부터 손대지"를 안 헤맨다.

**갱신 (2026-07-21) — 수요가 실제로 왔고, 추상화 레이어는 이미 생겼다.**
D2는 "중립 추상화 레이어는 넣지 않는다(YAGNI)"였는데, 도구통 훑기(ROADMAP Phase 4.7 ③)를
만들면서 `src/pouch/hosts/` 어댑터 층이 실제로 섰다. 미룬 게 아니라 **YAGNI 조건이 풀린
것** — 오너 컴퓨터에 Codex·Kiro가 실제로 깔려 있어 "언젠가"가 "지금"이 됐다.

- **지금 어댑터가 덮는 것** — 도구를 어디 두는지(`toolbox_paths`)와 훅 배선
  (`config_path`·기억 주입/사용 기록 install·remove). 두 갈래로 갈렸다:
  훅을 거는 하네스(Claude·Codex)와 파일로 거는 하네스(Kiro steering).
- **D2에서 살아남은 것** — *우선순위*는 여전히 Claude Code다(내장 세트·문서·기본 흐름의
  기준). 뒤집힌 건 "추상화를 안 만든다"뿐이지 "다 똑같이 대접한다"가 아니다.
- ~~**열린 질문**~~ — **✅ 닫힘 (2026-07-22, 오너 락).** 규칙(rule)처럼 하네스마다
  형식·자리가 다른 종류를 어떻게 담을지. 배승도가 훑기 작업 중 남긴 숙제("나머지
  rule 같은 걸 어떻게 정리할지 고민 좀 해봐야겠다", 2026-07-21)의 답:
  - **접었다 되편다** — 평면 장부엔 `<묶음>__<이름>`으로 접어 담고, 표면에 올릴 땐
    원래 폴더로 되편다. 오너의 말: *"절차가 하나 더 생기지만 그게 거짓말을 하는건
    아니니까"*. 접기·펴기는 한 파일(`catalog/docid.py`)에 같이 산다.
  - **자리가 없는 하네스엔 지어내지 않는다** — Codex·Kiro엔 규칙 자리가 확인된 바
    없어 표면은 Claude만 대상. D2에서 살아남은 "우선순위는 Claude"와 일관된다.
  - 구현·수리 세부는 ROADMAP Phase 4.7 ⑤ 참조.

## 예고된 문제 — 지금 안 하지만, 터질 자리를 알고 있는 것

### P1. boundary의 출처(provenance) — ✅ 해소 (2026-07-07)

필수 3과 함께 정면으로 설계·완료했다. 확정 그림:

- **direction 필드** (allow/ask/deny) — boundary 설계가 "deny 오독 터질 때 박는다"고
  예고한 그 필드를, drop을 방향으로 가르는 요구가 정당하게 당겼다. 산문에서 방향을
  뽑지 않고(오독 위험) 명시 라벨로만 읽는다. context 주입에 `[DENY]` 라벨로 강조.
- **source 필드** (`user` / `vendored:<도구id>`) — 사람이 건 것과 도구가 딸고 온 것을
  가른다. CLI로는 `user`만 가능(참칭 불가), `vendored:`는 승격 통로만 프로그램으로 새김.
- **승격 통로** — 엔트리의 권장 boundary(`RecommendedBoundary` 씨앗)가 설치 시 진짜
  boundary 메모리로 태어나며 출처 도장을 단다. 없을 때만 심음(재부착이 사용자 편집 안 뭉갬).
- **drop gate** — 도구 내릴 때 그 도구 출신 중 **allow는 함께 강등, ask·deny·방향불명은
  잔존+경고**. 사람이 건 것은 방향 무관 잔존. 안전 비대칭(허용 좁게·금지 넓게)과 일치.

### P2. 레지스트리 심볼릭 링크 — 신뢰를 넓힐 때 막을 자리 (2026-07-18, 오너 확인 락)

> ✅ **오너 확인(배승도, 2026-07-18).** 오너의 말: *"pouch가 set pull할때 도구
> 목록중 의도하지않은 동작을 하도록 명시한것이 잇으면 무시하도록 규칙을
> 짜자는거잔아 지금당장은 필요없고? public열떄 고민하면되니까"* — 예고만 남기고
> 지금은 안 고친다. 막는 시점은 public(외부·비개발자)으로 신뢰를 넓힐 때.

`pouch set pull`은 팀 공유 git 저장소를 통째로 복제한다. 그 안에 **심볼릭 링크**(다른
폴더를 가리키는 바로가기)가 들어있으면, 세트를 훑을 때 그 링크를 따라가 내 컴퓨터의
엉뚱한 파일까지 세트 목록으로 읽어올 수 있다. 지금은 대상이 "믿는 엔지니어 팀 내부
저장소"(RAFT-DESIGN 좁힌 락)라 위험이 낮고, 세트를 실제로 적용할 땐 사람 동의가 한 번
더 막는다. **비개발자·외부로 신뢰를 넓히는 순간** 막아야 한다 — 훑을 때 저장소 폴더
밖을 가리키는 링크는 따라가지 않고 건너뛰기.
